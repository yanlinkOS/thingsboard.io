This guide walks you through setting up ThingsBoard Community Edition in cluster mode using Docker Compose with microservices architecture.

For a simpler single-node installation, see <DocLink product={Products.CE} path="installation/docker">Docker (Linux, macOS)</DocLink>.

## Prerequisites

Install Docker:
- <a href="https://docs.docker.com/engine/install/ubuntu/" target="_blank">Docker for Ubuntu</a>
- <a href="https://docs.docker.com/engine/install/centos/" target="_blank">Docker for CentOS</a>
- <a href="https://docs.docker.com/desktop/setup/install/mac-install/" target="_blank">Docker Desktop for macOS</a>

A server with at least **8 GB of RAM** (16 GB recommended for production cluster deployments).

<Aside type="caution">
On Linux, add your user to the `docker` group so you can run commands without `sudo`.
See <a href="https://docs.docker.com/engine/install/linux-postinstall/#manage-docker-as-a-non-root-user" target="_blank">Manage Docker as a non-root user</a>.
</Aside>

## Step 3. Configure database

Edit the `.env` file to set the database type:

```bash
nano .env
```

Set the `DATABASE` variable to one of:

| Value | Description |
|---|---|
| `postgres` | Use PostgreSQL for all data |
| `hybrid` | Use PostgreSQL for entities and Cassandra for time-series data |

<Aside type="note">
The corresponding Docker services are defined in `docker-compose.postgres.yml` and `docker-compose.hybrid.yml`.
</Aside>

## Step 4. Choose queue service

ThingsBoard cluster deployments require an external message broker. In Memory queue is **not** suitable for cluster mode.

- **Kafka** — recommended for production. Used on most ThingsBoard production environments. Works for on-prem and private cloud deployments.
- **Confluent Cloud** — fully managed streaming platform based on Kafka. Useful for cloud-agnostic deployments.

<Tabs>
<TabItem label="Kafka (recommended)">

<a href="https://kafka.apache.org/" target="_blank">Apache Kafka</a> is an open-source stream-processing platform.

Edit the `.env` file:

```bash

```

Verify the following line:

```bash
TB_QUEUE_TYPE=kafka
```

## Step 5. Enable monitoring (optional)

Edit the `.env` file:

```bash
nano .env
```

Set `MONITORING_ENABLED` to `true`:

```bash
MONITORING_ENABLED=true
```

After deployment, Prometheus will be available at `http://localhost:9090` and Grafana at `http://localhost:3000` (default login: `admin` / `foobar`).

## Step 6. Install and start ThingsBoard

### Create host volumes

Create log folders for the services. The script requires `sudo` permissions to change ownership:

```bash
./docker-create-log-folders.sh
```

Verify that all required volume folders are available and have correct ownership:

```bash
./docker-check-log-folders.sh
```

<Aside type="caution" title="Docker Desktop on macOS">
If you use Docker Desktop on macOS with <a href="https://docs.docker.com/desktop/features/synchronized-file-sharing/" target="_blank">Synchronized file shares</a> (enabled by default for `/Users` subdirectories), skip the ownership change:

```bash
./docker-create-log-folders.sh --skipChown
./docker-check-log-folders.sh --skipChown
```
</Aside>

### Run installation

```bash
./docker-install-tb.sh --loadDemo
```

The `--loadDemo` flag loads sample tenant account, dashboards, and devices for evaluation and testing.

### Start services

```bash
./docker-start-services.sh
```

After a while when all services are started, open `http://localhost` in your browser. You should see the ThingsBoard login page. Use the following default credentials:

- **System Administrator**: sysadmin@thingsboard.org / sysadmin
- **Tenant Administrator**: tenant@thingsboard.org / tenant
- **Customer User**: customer@thingsboard.org / customer

You can change passwords for each account in the account profile page.

See <DocLink product={Products.CE} path="getting-started">Getting Started</DocLink> for your next steps after login.

## Inspect logs and control containers

Stream ThingsBoard node logs:

```bash
docker compose logs -f tb-core1 tb-rule-engine1
```

View the state of all containers:

```bash
docker compose ps
```

Stream logs of all services:

```bash
docker compose logs -f
```

Stop all services:

```bash
./docker-stop-services.sh
```

Remove all deployed containers:

```bash
./docker-remove-services.sh
```

Update specific services (pull newer image and rebuild container):

```bash
./docker-update-service.sh [SERVICE...]
```

If `[SERVICE...]` is omitted, all services are updated.

## Post-installation

### Generate HTTPS certificate

The deployment uses HAProxy for proxying traffic to containers. By default ports 80 and 443 are used. To use HTTPS with a valid certificate:

```bash
docker exec haproxy-certbot certbot-certonly --domain your_domain --email your_email
docker exec haproxy-certbot haproxy-refresh
```

<Aside type="note">
A valid certificate is used only when you access the Web UI by domain name. If you use an IP address, a self-signed certificate is used.
</Aside>

## Upgrading

When a new ThingsBoard release becomes available, update your installation to benefit from the latest features and security patches.

<Aside type="note">
To maintain database integrity and system stability, perform upgrades sequentially (e.g. v4.0.2 → v4.1.0 → v4.2.0).
</Aside>

See the <DocLink product={Products.CE} path="installation/upgrade-instructions/" target="_blank">Upgrade Instructions</DocLink> for detailed steps.

## Troubleshooting

### DNS issues

If you observe errors related to DNS issues, for example:

```bash
127.0.1.1:53: cannot unmarshal DNS message
```

Configure your system to use Google public DNS servers. See <a href="https://developers.google.com/speed/public-dns/docs/using#linux" target="_blank">Linux</a> and <a href="https://developers.google.com/speed/public-dns/docs/using#mac_os" target="_blank">macOS</a> instructions.
