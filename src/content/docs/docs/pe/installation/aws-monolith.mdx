This guide walks you through deploying ThingsBoard PE in monolith mode on AWS EKS. We use Amazon RDS for managed PostgreSQL.

## Prerequisites

### Install and configure tools

Install <a href="https://kubernetes.io/docs/tasks/tools/" target="_blank">kubectl</a>, <a href="https://docs.aws.amazon.com/eks/latest/userguide/eksctl.html" target="_blank">eksctl</a>, and <a href="https://docs.aws.amazon.com/cli/latest/userguide/install-cliv2.html" target="_blank">AWS CLI</a>.

## Step 2. Configure and create EKS cluster

In the `cluster.yml` file you can find the suggested cluster configuration. Key fields you can change:

| Field | Default | Description |
|---|---|---|
| `region` | `us-east-1` | AWS region for the cluster |
| `availabilityZones` | `[us-east-1a, us-east-1b, us-east-1c]` | Region availability zones |
| `instanceType` | `m5.xlarge` | EC2 instance type for nodes |

<Aside type="note">
If you don't change `instanceType` and `desiredCapacity`, EKS will deploy **1** node of type **m5.xlarge**.
</Aside>

<Aside type="note">
The following command creates a new VPC for your ThingsBoard cluster. You can also use an existing VPC and subnets. See <a href="https://eksctl.io/usage/vpc-networking/" target="_blank">eksctl VPC networking</a> for details.
</Aside>

Create the cluster:

```bash
eksctl create cluster -f cluster.yml
```

## Step 3. Create AWS load-balancer controller

Once the cluster is ready, create the AWS load-balancer controller by following <a href="https://docs.aws.amazon.com/eks/latest/userguide/aws-load-balancer-controller.html" target="_blank">this guide</a>.

The cluster provisioning scripts create several load balancers:

| Load Balancer | Type | Purpose |
|---|---|---|
| `tb-http-loadbalancer` | ALB | Web UI, REST API, HTTP transport |
| `tb-mqtt-loadbalancer` | NLB | MQTT transport |
| `tb-coap-loadbalancer` | NLB | CoAP transport |
| `tb-edge-loadbalancer` | NLB | Edge instances connectivity |

<Aside type="caution">
Provisioning the AWS load-balancer <a href="https://docs.aws.amazon.com/eks/latest/userguide/aws-load-balancer-controller.html" target="_blank">controller</a> is a **critical step** required for the load balancers to work properly.
</Aside>

## Step 4. Provision databases

### 4.1 Amazon PostgreSQL DB configuration

Set up PostgreSQL on Amazon RDS. ThingsBoard uses it as the main database for devices, dashboards, rule chains, and device telemetry. Follow <a href="https://docs.aws.amazon.com/AmazonRDS/latest/UserGuide/CHAP_GettingStarted.CreatingConnecting.PostgreSQL.html" target="_blank">this guide</a>, but take into account the following requirements:

- Keep your PostgreSQL password in a safe place. We will refer to it later as **YOUR_RDS_PASSWORD**.
- Make sure your PostgreSQL version is latest **16.x**.
- Make sure your PostgreSQL RDS instance is accessible from the ThingsBoard cluster. The easiest way is to deploy the RDS instance in the same VPC and use the `eksctl-thingsboard-cluster-ClusterSharedNodeSecurityGroup-*` security group.
- Make sure you use **"thingsboard"** as the initial database name. If you do not specify a database name, Amazon RDS does not create one.

Recommendations:
- Use **Production** template for high availability.
- Use **Provisioned IOPS** for better performance.
- Consider creating a custom parameters group for your RDS instance.
- Consider deploying the RDS instance into private subnets.

### 4.2 Cassandra (optional)

Using Cassandra is optional. We recommend it if you plan to insert more than 5K data points per second or want to optimize storage space.

#### Provision additional node groups

Provision additional node groups to host Cassandra instances. At least 4 vCPUs and 16 GB of RAM is recommended.

Create **3** separate node pools with **1** node per zone. Since we use EBS disks, we don't want K8S to launch a pod in a zone where the corresponding disk is unavailable.

```bash
eksctl create nodegroup --config-file=<path> --include='cassandra-*'
```

#### Deploy Cassandra stateful set

```bash
kubectl apply -f tb-namespace.yml
kubectl config set-context $(kubectl config current-context) --namespace=thingsboard
```

Deploy Cassandra:

```bash
kubectl apply -f receipts/cassandra.yml
```

The startup may take a few minutes. Monitor the process:

```bash
kubectl get pods
```

#### Update DB settings

Edit the ThingsBoard DB settings file and add Cassandra settings. Don't forget to replace **YOUR_AWS_REGION** with the name of your AWS region.

```bash
echo "  DATABASE_TS_TYPE: cassandra" >> tb-node-db-configmap.yml
echo "  CASSANDRA_URL: cassandra:9042" >> tb-node-db-configmap.yml
echo "  CASSANDRA_LOCAL_DATACENTER: YOUR_AWS_REGION"  >> tb-node-db-configmap.yml
```

Verify:

```bash
cat tb-node-db-configmap.yml | grep DATABASE_TS_TYPE
```

Expected output: `DATABASE_TS_TYPE: cassandra`

#### Create keyspace

```bash
kubectl exec -it cassandra-0 -- bash -c "cqlsh -e \
    \"CREATE KEYSPACE IF NOT EXISTS thingsboard \
    WITH replication = { \
        'class' : 'NetworkTopologyStrategy', \
        'us-east' : '3' \
    };\""
```

## Step 8. Configure load balancers

### 8.1 Configure HTTP(S) load balancer

You have 2 options:

- **HTTP** — recommended **for development**. Simple configuration and minimum costs.
- **HTTPS** — recommended **for production**. Acts as an SSL termination point with automatic redirect from HTTP to HTTPS.

#### HTTP load balancer

```bash
kubectl apply -f receipts/http-load-balancer.yml
```

Check the status:

```bash
kubectl get ingress
```

Once provisioned, you should see:

```text
NAME                   CLASS    HOSTS   ADDRESS         PORTS   AGE
tb-http-loadbalancer   <none>   *       34.111.24.134   80      7m25s
```

Use the address to access the HTTP web UI (port 80) and connect devices via <DocLink product={Products.PE} path="reference/http-api">HTTP API</DocLink>.

Default credentials:
- **System Administrator**: sysadmin@thingsboard.org / sysadmin
- **Tenant Administrator**: tenant@thingsboard.org / tenant (if demo data loaded)
- **Customer User**: customer@thingsboard.org / customer (if demo data loaded)

#### HTTPS load balancer

Use <a href="https://aws.amazon.com/certificate-manager/" target="_blank">AWS Certificate Manager</a> to create or import an SSL certificate. Note your certificate ARN.

Edit the load balancer configuration and replace **YOUR_HTTPS_CERTIFICATE_ARN** with your certificate ARN:

```bash
nano receipts/https-load-balancer.yml
```

Deploy the HTTPS load balancer:

```bash
kubectl apply -f receipts/https-load-balancer.yml
```

### 8.2 Configure MQTT load balancer (optional)

Configure MQTT load balancer if you plan to use MQTT protocol to connect devices.

```bash
kubectl apply -f receipts/mqtt-load-balancer.yml
```

The load balancer forwards all TCP traffic for ports 1883 and 8883.

#### One-way TLS

The simplest way to configure MQTTS is to make the AWS NLB act as a TLS termination point. Traffic between devices and the load balancer is encrypted, while traffic between the load balancer and MQTT Transport is not encrypted. There should be no security issues since the NLB runs in your VPC.

<Aside type="note">
The main disadvantage of this option is that you can't use "X.509 certificate" MQTT client credentials, since certificate information is not transferred from the load balancer to the ThingsBoard MQTT Transport service.
</Aside>

Use <a href="https://aws.amazon.com/certificate-manager/" target="_blank">AWS Certificate Manager</a> to create or import an SSL certificate. Edit the load balancer configuration:

```bash
nano receipts/mqtts-load-balancer.yml
```

Replace **YOUR_MQTTS_CERTIFICATE_ARN** with your certificate ARN, then deploy:

```bash
kubectl apply -f receipts/mqtts-load-balancer.yml
```

#### Two-way TLS

Follow the <DocLink product={Products.PE} path="user-guide/mqtt-over-ssl">MQTT over SSL</DocLink> guide to create a `.pem` file. Store it as `server.pem` in the working directory.

Create a config-map with your PEM file:

```bash
kubectl create configmap tb-mqtts-config \
 --from-file=server.pem=YOUR_PEM_FILENAME \
 --from-file=mqttserver_key.pem=YOUR_PEM_KEY_FILENAME \
 -o yaml --dry-run=client | kubectl apply -f -
```

Uncomment all sections in the `tb-node.yml` file marked with "Uncomment the following lines to enable two-way MQTTS", then apply:

```bash
kubectl apply -f tb-node.yml
```

Deploy the "transparent" load balancer:

```bash
kubectl apply -f receipts/mqtt-load-balancer.yml
```

### 8.3 Configure UDP load balancer (optional)

Configure UDP load balancer if you plan to use CoAP or LwM2M protocol.

```bash
kubectl apply -f receipts/udp-load-balancer.yml
```

The load balancer forwards all UDP traffic for ports:

| Port | Protocol |
|---|---|
| 5683 | CoAP non-secure |
| 5684 | CoAP secure DTLS |
| 5685 | LwM2M non-secure |
| 5686 | LwM2M secure DTLS |
| 5687 | LwM2M bootstrap DTLS |
| 5688 | LwM2M bootstrap secure DTLS |

For CoAP over DTLS, follow the <DocLink product={Products.PE} path="user-guide/coap-over-dtls">CoAP over DTLS</DocLink> guide. For LwM2M over DTLS, follow the <DocLink product={Products.PE} path="user-guide/ssl/lwm2m-over-dtls">LwM2M over DTLS</DocLink> guide.

### 8.4 Configure Edge load balancer (optional)

Configure the Edge load balancer if you plan to connect Edge instances to your ThingsBoard server.

```bash
kubectl apply -f receipts/edge-load-balancer.yml
```

The load balancer forwards all TCP traffic on port 7070.

To get the external IP address:

```bash
kubectl get services | grep "EXTERNAL-IP\|tb-edge-loadbalancer"
```

Use the external IP address as **CLOUD_RPC_HOST** in Edge connection parameters.

## Upgrading

### Upgrading to new ThingsBoard version

Merge your local changes with the latest release branch from the repo you cloned in [Step 1](#step-1-clone-thingsboard-pe-k8s-scripts-repository).

If a database upgrade is needed:

```bash
./k8s-upgrade-tb.sh --fromVersion=[FROM_VERSION]
```

Where `FROM_VERSION` is the starting version. See <DocLink product={Products.PE} path="installation/upgrade-instructions/" target="_blank">Upgrade Instructions</DocLink> for valid `fromVersion` values. You must upgrade versions one by one (e.g., 3.6.1 → 3.6.2 → 3.6.3).

<Aside type="note">
You may optionally stop the tb-node pods while running the database upgrade. This causes downtime but ensures DB state consistency. Most updates do not require stopping the pods.
</Aside>

Once completed, re-deploy resources:

```bash
./k8s-deploy-resources.sh
```

### Upgrading to new Trendz version (optional)

Pull the latest changes:

```bash
git pull origin master
```

Then execute:

```bash
./k8s-upgrade-trendz.sh
```

<Aside type="note">
You can upgrade Trendz to the latest version from any previous version (e.g., 1.12.0 → 1.15.0).
</Aside>

## Cluster deletion

Delete all ThingsBoard pods:

```bash
./k8s-delete-resources.sh
```

Delete all ThingsBoard pods and configmaps:

```bash
./k8s-delete-all.sh
```

Delete the EKS cluster (change cluster name and region as needed):

```bash
eksctl delete cluster -r us-east-1 -n thingsboard -w
```
