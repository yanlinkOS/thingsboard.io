Trendz treats your ThingsBoard entity hierarchy as a flat analytical table. Assets, Devices, and their attributes and telemetry become columns; relations between entities act as joins. You pick the fields you need, and Trendz resolves the full topology automatically.

## Entity topology

Consider a smart building solution where Buildings contain Apartments, and Apartments contain Energy Meters:

```plaintext
  ┌──────────────┐
  │   Building   │  ← Asset profile
  └──────┬───────┘
         │ contains
  ┌──────┴───────┐
  │  Apartment   │  ← Asset profile
  └──────┬───────┘
         │ contains
  ┌──────┴───────┐
  │ Energy Meter │  ← Device profile
  └──────────────┘
```

Trendz flattens this tree into a virtual table with columns for every attribute and telemetry key across all levels. When you build a view that references `building name` and `energy` telemetry, Trendz walks the relation chain to match each meter to its building.

## How data resolution works

When you create a report using `building name` (from the Building asset) and `energy` telemetry (from the Energy Meter device) with `SUM` aggregation over the last month, Trendz executes the following sequence:

<Steps>

1. Find all Building assets in ThingsBoard.

2. Resolve all Apartments related to each Building.

3. Resolve all Energy Meters related to each Apartment.

4. Load `energy` telemetry for the last month from every resolved Energy Meter.

5. Aggregate the telemetry per building using `SUM`. The result shows total energy consumed by each building.

</Steps>

<Aside type="tip">
You do not need to write queries or manage joins. Select the fields you need in the Trendz view builder, and the platform resolves the topology behind the scenes.
</Aside>

## Managing relations between entity profiles

ThingsBoard allows multiple relation types between the same entities. Trendz requires **exactly one active relation type between any two device or asset profiles**. When multiple relations exist between the same profiles, Trendz cannot determine which path to follow, resulting in undefined behavior.

**Example:** Apartment → EnergyMeter has both `contains` and `connected` relations. Trendz does not know which relation to traverse when resolving meters for a building.

**Fix:** Disable the unwanted relation in the Trendz Business Entity configuration.

<Steps>

1. Click the Business Entity button in the bottom-left corner of the Trendz UI.

2. Select the entity profile with conflicting relations (like `EnergyMeter` or `Apartment`).

3. Open the **Relations** tab.

4. Uncheck **Enabled** for the relation you want to exclude.

5. Click **Save**.

</Steps>
