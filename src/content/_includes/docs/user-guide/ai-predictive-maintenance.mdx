This guide builds an end-to-end IoT anomaly detection pipeline: devices send vibration, temperature, and acoustic data → <DocLink product={props.product} path="user-guide/calculated-fields">Calculated Fields</DocLink> maintain a rolling window → an <DocLink product={props.product} path="reference/rule-engine/nodes/external/ai-request">AI request</DocLink> rule node classifies anomalies → ThingsBoard creates alarms and sends notifications.

## Prerequisites

- ThingsBoard version **4.2+**.
- Device(s) capable of sending the required telemetry values (this guide emulates them with `curl`).
- LLM provider credentials (OpenAI, Azure OpenAI, etc.) — see <DocLink product={props.product} path="user-guide/ai-models">AI models</DocLink>.

## Telemetry input and output

**Expected telemetry keys (per device):**

| Key | Type | Description |
|---|---|---|
| `vibration` | float | Vibration level in mm/s |
| `temperature` | float | Temperature in °C |
| `acousticDev` | float | Acoustic deviation from baseline in % |

**AI output (JSON):**

```json
{
  "anomaly": "Bearing Wear",
  "summary": "Vibration has reached 7.4 mm/s and temperature is at 86 °C accompanied by irregular acoustic patterns, indicating bearing wear. Recommend immediate bearing inspection and replacement to avoid catastrophic failure."
}
```

**Important details:**

- System and user prompts can reference incoming message data using <DocLink product={props.product} path="reference/rule-engine/templatization">templatization</DocLink>:
  - `$[*]` — entire message body
  - `${'{*}'}` — entire message metadata
  - `$[key]` — a specific message body field
  - `${'{key}'}` — a specific metadata value
- Supported response formats: **TEXT**, **JSON**, and **JSON Schema** (this guide uses JSON Schema).
- A <DocLink product={props.product} path="reference/rule-engine/nodes/transformation/deduplication">deduplication</DocLink> node with a 5-second interval is used to reduce AI token usage.

## Performance and cost controls

API costs are controlled by the provider's API key and billing — ThingsBoard does not track usage or enforce limits internally. To manage costs effectively:

- **Deduplication** — use the <DocLink product={props.product} path="reference/rule-engine/nodes/transformation/deduplication">deduplication</DocLink> node to batch incoming messages (e.g., 5-second window) so the AI is called once per batch, not per telemetry message.
- **Early filters** — skip the AI call entirely if all metrics are within normal bands. Add a <DocLink product={props.product} path="reference/rule-engine/nodes/filter/script">script filter</DocLink> node before the AI request node.
- **Payload compacting** — send statistics (min, mean, max, standard deviation, trend) instead of full arrays if the prompt allows.
- **Model selection** — smaller models (e.g., `gpt-4o-mini`) are significantly cheaper per token than large models while still providing good anomaly classification.

## Troubleshooting

| Problem | Steps |
|---|---|
| **No alarms created** | 1. Verify raw device data appears in the **Latest telemetry** tab. 2. Enable and review <DocLink product={props.product} path="user-guide/calculated-fields">calculated field</DocLink> debug events. 3. Enable and review the corresponding rule node debug events. |
| **High costs** | Increase the deduplication period, tune the prompt to use fewer tokens, or switch to a cheaper AI model. |
