Kafka Integration connects ThingsBoard to <a href="https://kafka.apache.org/" target="_blank">Apache Kafka</a> message brokers. It subscribes to one or more Kafka topics, consumes incoming messages, and transforms them into ThingsBoard telemetry and attributes via an uplink converter.

<DocImage src="/src/assets/images/user-guide/integrations/kafka/Kafka_main.png" alt="Kafka Integration architecture diagram — devices and IoT platforms publish messages to Kafka brokers; the Kafka Integration consumes topics, passes payloads through an uplink converter, and pushes telemetry and attributes to ThingsBoard; a separate Kafka Rule Node handles downlink" />

## Prerequisites

A running Kafka broker is required before creating the integration.

Use a local installation or a cloud-managed cluster such as <a href="https://www.confluent.io/" target="_blank">Confluent Cloud</a>.

## ThingsBoard integration setup

### Create an uplink converter

The uplink converter decodes the Kafka message body and maps it to the ThingsBoard data model. Kafka uses a <DocLink product={props.product} path="user-guide/integrations/uplink-data-converter">generic uplink converter</DocLink>.

The decoder function receives:
- `payload` — raw bytes of the Kafka message value
- `metadata` — `integrationName` and any key-value pairs configured in the integration's Metadata settings

It must return `deviceName`, `deviceType`, `telemetry`, and `attributes`. Both `telemetry` and `attributes` are flat key-value maps — nested objects are not supported.

<Aside type="tip">
  Use <DocLink product={props.product} path="user-guide/tbel">TBEL</DocLink> for converter scripts — it executes significantly faster than plain JavaScript inside ThingsBoard.
</Aside>

In this example, device **43T1YH-REE** sends a JSON payload where telemetry readings are packed as a 4-byte hex string in the `data` field:

```json
{
  "EUI": "43T1YH-REE",
  "ts": 1638876127000,
  "data": "3d1f0059",
  "port": 10,
  "freq": 24300,
  "rssi": -130,
  "serial": "230165HRT"
}
```

`EUI` is used as the device name. Each 2-character hex segment of `data` encodes one sensor reading:
- `3d` &#8702; Temperature
- `1f` &#8702; Humidity
- `00` &#8702; Fan speed
- `59` &#8702; Pressure

}

/** Helper functions **/

function decodeToString(payload) {
    return String.fromCharCode.apply(String, payload);
}

function decodeToJson(payload) {
    var str = decodeToString(payload);
    return JSON.parse(str);
}

function hexToInt(value) {
    return parseInt('0x' + value.match(/../g).reverse().join(''));
}

#### Connection settings

<h4>Group ID</h4>

A string you define — it identifies the consumer group this integration belongs to. You can use any value (e.g. `tb-kafka-group`). All integration instances sharing the same Group ID form a consumer group and split the partition load, so each message is delivered to only one instance.

<h4>Client ID</h4>

A string you define — it labels this consumer client for monitoring and logging purposes (e.g. `tb-kafka-client`). Must be unique per running consumer if you need to distinguish instances in Kafka metrics.

<h4>Bootstrap servers</h4>

Comma-separated list of Kafka broker addresses in `host:port` format, e.g. `localhost:9092`. Used to establish the initial connection to the cluster.

<h4>Poll interval (ms)</h4>

How often the integration polls Kafka for new messages. Default: `5000` ms.

<h4>Topics</h4>

Comma-separated list of Kafka topics to subscribe to, e.g. `my-topic`.

<h4>Auto create topics</h4>

When enabled, ThingsBoard automatically creates any subscribed topics that do not already exist on the broker.

<h4 id="other-properties">Other properties</h4>

Additional Kafka consumer properties as key-value pairs. Use this for security configuration such as SASL/SSL when connecting to Confluent Cloud or other secured brokers.

For **Confluent Cloud**, add the following four entries:

| Key | Value |
|-----|-------|
| `ssl.endpoint.identification.algorithm` | `https` |
| `sasl.mechanism` | `PLAIN` |
| `sasl.jaas.config` | `org.apache.kafka.common.security.plain.PlainLoginModule required username="CLUSTER_API_KEY" password="CLUSTER_API_SECRET";` |
| `security.protocol` | `SASL_SSL` |

Replace `CLUSTER_API_KEY` and `CLUSTER_API_SECRET` with the API key and secret from your Confluent Cloud cluster's **Data integration → API keys** section.

<h4>Execute remotely</h4>

When enabled, ThingsBoard generates an **Integration key** and **Integration secret**. These credentials allow the integration to run as a separate process outside the ThingsBoard cluster — useful for deployments where the integration must reside in a DMZ or on-premises gateway.

## Downlink

The Kafka Integration does not have a dedicated downlink converter. To publish messages from ThingsBoard back to a Kafka topic, use the <DocLink product={props.product} path="reference/rule-engine/nodes/external/kafka">Kafka Rule Node</DocLink> in the rule engine.

<Aside type="caution">
Do not subscribe the integration and the Kafka Rule Node to the **same topic** — this creates a message loop where each consumed message triggers another publish to the same topic, causing runaway processing.
</Aside>

Configure the Kafka Rule Node with the same bootstrap servers and security properties as the integration, but point it to a **separate downlink topic**.
