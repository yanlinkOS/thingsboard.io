**Azure Service Bus** is a fully managed enterprise message broker from Microsoft that supports queues and publish/subscribe topics. The ThingsBoard Azure Service Bus integration subscribes to a topic on your Service Bus namespace, receives device messages, decodes them via an uplink converter, and stores telemetry and attributes in ThingsBoard. In the reverse direction, it encodes Rule Engine messages via a downlink converter and publishes them to a dedicated downlink topic.

<DocImage src="/src/assets/images/user-guide/integrations/azure-service-bus/azure-service-bus-overview.png" alt="Azure Service Bus integration overview diagram" />

## Configure Azure Service Bus

Before creating the integration in ThingsBoard, prepare the following in the <a href="https://portal.azure.com" target="_blank" rel="noopener noreferrer">Azure portal</a>:

<Steps>
1. Open your **Service Bus namespace** (or create one). A namespace is the scoping container for all messaging components and has a globally unique DNS name in the form `<namespace>.servicebus.windows.net`.
2. Create a **topic** for uplink messages (e.g. `uplinktopic`). Topics implement the publish/subscribe pattern — multiple subscribers can independently consume the same messages.
3. Create a **subscription** on that topic (e.g. `S1`). Each subscription maintains its own independent copy of every message published to the topic, so ThingsBoard consumes messages without affecting other consumers.
4. Copy the **connection string** for the namespace — go to **Settings &#8702; Shared access policies**, open a policy (e.g. `RootManageSharedAccessKey`), and copy the **Primary Connection String**. It has the form `Endpoint=sb://<namespace>.servicebus.windows.net/;SharedAccessKeyName=<policy>;SharedAccessKey=<key>`.
</Steps>

<Aside type="note">
  For downlink support, also create a dedicated downlink topic (e.g. `downlinktopic`) with its own subscription before configuring the integration.
</Aside>

## ThingsBoard integration setup

### Create an uplink converter

The uplink converter decodes incoming Azure Service Bus messages and maps them to the ThingsBoard data model. Azure Service Bus uses a <DocLink product={props.product} path="user-guide/integrations/uplink-data-converter">generic uplink converter</DocLink>.

**Sample input message:**

```json
{
  "devName": "Sensor A1",
  "msg": {
    "temp": 23,
    "humidity": 40
  }
}
```

The `devName` field maps to `deviceName`. The nested `msg` object provides `temperature` and `humidity` telemetry.

function decodeToString(payload) {
    return String.fromCharCode.apply(String, payload);

    return data;

#### Connection settings

<h4>Connection string</h4>

The Azure Service Bus namespace connection string used to authenticate the integration. Found under **Settings &#8702; Shared access policies** in your namespace — copy the **Primary Connection String** of any policy that has `Listen` permission (e.g. `RootManageSharedAccessKey`). Format: `Endpoint=sb://<namespace>.servicebus.windows.net/;SharedAccessKeyName=<policy>;SharedAccessKey=<key>`.

<h4>Topic name</h4>

The name of the Azure Service Bus topic ThingsBoard subscribes to for uplink messages (e.g. `uplinktopic`).

<h4>Subscription name</h4>

The name of the subscription on the uplink topic that ThingsBoard consumes (e.g. `S1`). Each subscription receives an independent copy of every message published to the topic.

<h4>Execute remotely</h4>

When enabled, ThingsBoard generates an **Integration key** and **Integration secret** that allow the integration to run as a separate process outside the ThingsBoard cluster — useful when the Azure Service Bus endpoint is only reachable from a specific network.

<h4>Advanced settings</h4>

- **Downlink connection string** — the connection string for the namespace where the downlink topic resides (can be the same namespace as the uplink); requires `Send` permission
- **Downlink topic name** — the name of the Azure Service Bus topic ThingsBoard publishes downlink messages to (e.g. `downlinktopic`)
- **Description** — optional text description for this integration
- **Metadata** — key-value pairs injected into every uplink message as `integrationMetadata` in converter scripts

</Steps>

## Configure downlink

To send messages from ThingsBoard back to Azure Service Bus, create a dedicated downlink topic in Azure, add a downlink converter, configure the integration's advanced settings, and wire the Root Rule Chain to forward attribute updates.

### Create the downlink converter

The downlink converter encodes Rule Engine messages into Azure Service Bus payloads. The output must include:
- `contentType` — how data is encoded: `TEXT`, `JSON`, or `BINARY`
- `data` — the payload to publish to the downlink topic
- `metadata.deviceId` — identifier passed as a message property to route the message to the correct device

### Attach the downlink converter and configure the downlink topic

<Steps>
1. Go to **Integrations center &#8702; Integrations** and open the **Azure Service Bus** integration.
2. Click the edit (pencil) icon.
3. In the **Downlink data converter** field, select `Service Bus Downlink Converter`.
4. Expand **Advanced settings** and fill in:
    - **Downlink connection string** — the namespace connection string (copy from the uplink settings if you use the same namespace)
    - **Downlink topic name** — the name of the downlink topic (e.g. `downlinktopic`)
5. Click **Apply changes**.
</Steps>

### Update the Root Rule Chain

Add an **Integration Downlink** node to the Root Rule Chain to forward attribute updates to Azure Service Bus.

<Steps>
1. Go to **Rule chains** and open the **Root Rule Chain**. Click the edit icon.
2. Find **Integration Downlink** in the node panel and drag it onto the canvas.
3. In the **Add rule node** dialog, enter a name (e.g. `Service Bus downlink node`), select **Azure Service Bus Integration**, and click **Add**.
4. Connect the **Message Type Switch** node to the **Integration Downlink** node using both the **Post attributes** and **Attributes Updated** relations.
5. Click **Apply changes**.
</Steps>
