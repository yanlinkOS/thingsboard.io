A **Widget** is a dashboard element that displays a specific type of data or provides specific functionality. Widgets enable visualization of device data, remote device control, alarm management, and display of custom HTML content.

Widgets are organized into [**widget bundles**](/docs/pe/user-guide/widgets/#widget-bundles), which group related widgets by functionality or use case.

---

## Widgets Library

The Widgets Library contains the complete ThingsBoard widget collection, organized by bundle and use case — data visualization, alarms, device control, SCADA, maps, industrial monitoring, and more.

## Add Widget to Dashboard

<Steps>
1. Open a dashboard and switch to **edit mode**.
2. Click **+ Add widget**.
3. Select a **widget bundle**, then choose the specific widget.
4. Configure the **datasource** — select an <DocLink product={props.product} path="user-guide/aliases" bold={true} target="_blank">entity alias</DocLink> (recommended) or select entities directly and add the required data keys.
5. Adjust **appearance** and **behavior** (title, units, colors, thresholds, actions, etc.).
6. Click **Add**.
7. Drag to the desired position and resize on the grid.
8. Click **Save** to persist changes and exit edit mode.

### Datasources

A datasource defines where a widget gets its data — which entities to query and which keys to read. Most widgets have one or more datasources; **Static** widgets have none. Each datasource specifies a **type**, an **entity scope** (a single device or an alias-resolved set), optional **filters**, and a list of **data keys**.

<Aside type="note">
  For **Control** widgets, the datasource does not read data — it identifies the **target device** to which RPC commands are sent. The data key list is replaced by RPC command configuration specific to each control widget.
</Aside>

#### Data Keys

A data key identifies the telemetry, attribute, or entity field a widget reads from its datasource. Each datasource can have multiple data keys.

**Key source types:**
- <DocLink product={props.product} path="user-guide/digital-twins/time-series-data/" bold={true} target="_blank">Time series</DocLink> — telemetry keys reported by devices or written via the Rule Engine/REST API.
- <DocLink product={props.product} path="user-guide/digital-twins/attributes/" bold={true} target="_blank">Attributes</DocLink> — Client, Server, and Shared attributes of the selected entity.
- **Entity fields** — metadata fields available for the entity type (e.g., name, type, label, created time).

Available sources depend on the widget type: time series widgets use time series keys; latest values widgets support all three types; alarm widgets also support alarm-specific fields; control and static widgets do not use data keys.

#### Data Filters

Filters narrow which data a widget retrieves from its data sources. Build a filter by adding keys and defining conditions on those keys — multiple keys and conditions can be combined.

See <DocLink product={props.product} path="user-guide/filters/" target="_blank">Filters</DocLink> for details.

#### Alarm Filters

Alarm widgets use a dedicated filter panel instead of a data key list. Use these filters to control which alarms appear in the widget — by status, severity, type, assignee, or entity scope.

- **Alarm status list** — show only alarms in the selected statuses: *Active*, *Cleared*, *Acknowledged*, *Unacknowledged*. Multiple statuses can be combined.
- **Alarm severity list** — show only alarms at the selected severities: *Critical*, *Major*, *Minor*, *Warning*, *Indeterminate*. Multiple severities can be combined.
- **Alarm type list** — restrict to specific alarm type names (e.g., `HighTemperature`). Leave empty to show all types.
- **Assignee** — filter by the user the alarm is assigned to, or leave as *All*.
- **Search propagated alarms** — when enabled, the widget includes alarms that originated on **child entities** and were propagated up to the parent. Useful when the datasource resolves an asset or group and you want to see alarms from all devices within it, not just direct alarms on the asset itself.

### Appearance

The Appearance tab controls the visual presentation, layout structure, and display behavior of a widget — how data is rendered and formatted — without affecting datasources, queries, or processing logic.

Available options depend on the widget type:

- **Value formatting** — default units, decimal precision, date format, and the "No data" message displayed when no data is available for the selected time window.
- **Layout structure** — layout type, auto scale, element visibility (label, icon, value, date), and icon sizing.
- **Chart rendering** — axes configuration (position, min/max, units), additional Y-axes, thresholds, grid lines, stacking, zoom, legend behavior, tooltip configuration, and aggregation display.
- **Table presentation** — header visibility, search, column management, actions column, pagination, tabs, and row styling.
- **Styling and theming** — background color, blur, padding, color palette, icon/value/label colors, and custom CSS.
- **Animation and visual states** — animation enablement, state-based styling, and dynamic effects.
- **Domain-specific visuals** (e.g., SCADA widgets) — symbol selection, flow direction, presence indicators, and animation speed.

### Widget Card

The widget card is the outer frame wrapping every widget on a dashboard. It provides a title bar, card-level visual styling, and optional header buttons available to end-users.

<h4>Card Title</h4>

- **Display widget title** — toggle that shows or hides the title bar entirely.
- **Title** — label displayed in the header. Supports dynamic patterns such as `${entityName}` or `${entityLabel}`.
- **Title tooltip** — tooltip text shown when hovering over the title.
- **Display title icon** — toggle that adds an icon before the title text. When enabled, configure the icon, its size, alignment, and color.
- **Advanced title style** — expandable section for fine-grained font and text styling (size, weight, color, decoration).

### Widget Actions

Actions connect user interactions to navigation or device control operations. Each action is defined by an **action source** (what triggers it) and an **action type** (what it does).

<h4>Action Sources</h4>

Action sources vary by widget type.

| Source | Widgets |
|---|---|
| **Widget header button** — a custom button in the widget title bar | All |
| **On row click** — clicking a table row | Tables |
| **On row double click** — double-clicking a table row | Tables |
| **Action cell button** — a custom button column in a table | Tables |
| **On HTML element click** — a tagged `id`-marked HTML element | HTML widgets |
| **On map element click** — markers and polygons | Map widgets |

<h4>Action Types</h4>

Action types are the same for all widgets.

| Type | Description |
|---|---|
| **Navigate to new dashboard state** | Open a named state in the current dashboard |
| **Update current dashboard state** | Update the entity context of the current state |
| **Navigate to other dashboard** | Open a different dashboard |
| **Open URL** | Open a web URL |
| **Custom action** | Execute a custom JavaScript function |
| **Custom action (with HTML template)** | Execute a function inside an HTML popup |
| **Mobile action** | Trigger a mobile device function (camera, QR scan, location) |
| **Place map item** | Create and place a new entity on the map *(map widgets only)* |

<h4>Add Action</h4>

**In basic mode** — open widget settings, scroll to **Actions**, click **Add action**, configure the source and type, then click **Add** and **Save**.

**In advanced mode** — open widget settings, go to the **Actions** tab, click **+**, configure, then click **Add** and **Apply**.

### Widget Layout Settings

The **Layout** tab controls how the widget behaves on the dashboard grid and on mobile screens.

<h4>Resize Options</h4>

- **Resizable** — when enabled, the widget can be resized by dragging its edges or corners in edit mode. Disable to lock the widget to its current size.
- **Preserve aspect ratio** — when enabled, resizing maintains the original width-to-height ratio.

<h4>Mobile</h4>

Controls widget visibility and sizing on mobile screens:

- **Hide widget in mobile mode** — removes the widget from the mobile layout while keeping it on desktop.
- **Hide widget in desktop mode** — removes the widget from the desktop layout while keeping it on mobile.
- **Order** — position of the widget in the mobile layout stack. Widgets stack vertically on mobile; lower values appear first.
- **Height** — widget height in the mobile grid (in grid units). Overrides the desktop height for mobile screens.

## Duplicate Widget

Duplicating a widget allows you to reuse an existing configuration instead of creating a new widget from scratch. This is useful when multiple widgets share the same layout, style, or behavior but use different data.

<h4>Create a Full Copy</h4>

A full copy creates an independent duplicate with all settings preserved.

<Steps>
  1. Open the dashboard in <strong>edit mode</strong>.
  2. Right-click the widget and select <strong>Copy</strong> (or press <strong>Ctrl+C</strong>).
  3. Open the target dashboard (if needed).
  4. Right-click an empty area of the canvas and select <strong>Paste</strong> (or press <strong>Ctrl+V</strong>).
  5. Reposition the widget and <strong>save dashboard</strong>.
</Steps>

A full copy can be pasted into the same dashboard or into a different dashboard. After duplication, the new widget is independent and can be modified without affecting the original.

<h4>Create Referenced Copy</h4>

If you are working with multiple dashboard layouts (<DocLink product={props.product} path="user-guide/layouts/#breakpoints" bold={true} target="_blank">breakpoints</DocLink>) and want configuration changes to sync across all of them, use widget references.
A referenced copy shares the same configuration as the original widget. Any changes made to one instance are automatically reflected in the other.

<Steps>
  1. Right-click the widget and select <strong>Copy reference</strong> (or press <strong>Ctrl+R</strong>).
  2. In the same dashboard, switch to another state.
  3. Right-click the canvas and select <strong>Paste reference</strong> (or press <strong>Ctrl+I</strong>).
  4. Reposition the widget and <strong>save dashboard</strong>.
</Steps>

Referenced copies can only be used within the same dashboard, typically across different states. They cannot be pasted into another dashboard.
