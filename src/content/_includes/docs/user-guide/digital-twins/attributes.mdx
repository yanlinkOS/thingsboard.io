**Attributes** are key-value pairs that describe the properties, state, or configuration of an entity. Unlike telemetry, attributes represent semi-static data — ThingsBoard stores only the **latest value** and its timestamp.

<Aside type="tip">
Use [camelCase](https://en.wikipedia.org/wiki/Camel_case) for attribute names. This simplifies writing JavaScript functions in the Rule Engine and dashboards.
</Aside>

## Data Model

Each attribute entry consists of:
- **Key** — the attribute name (e.g., `maxTemperatureThreshold`)
- **Value** — string, boolean, integer, double, or JSON object
- **Timestamp** — Unix epoch in milliseconds; the last time this attribute was updated

**Example payload:**
```json
{ "reportingInterval": 60, "firmwareVersion": "v2.4.0", "alarmEnabled": true }
```

Unlike <DocLink product={props.product} path='user-guide/digital-twins/time-series-data'>time-series data</DocLink>, only the **latest value per key** is retained — there is no history.

## Attribute Scopes

ThingsBoard supports three attribute scopes that define **who can read and write** each attribute:

| Scope | Writable by | Readable by | Available for |
|-------|-------------|-------------|---------------|
| Server-side | Platform only | Platform | All entities |
| Shared | Platform only | Platform + Device | Devices only |
| Client-side | Device only | Platform (read-only) | Devices only |

### Server-side attributes

```
┌──────────┐  ✗ no access  ┌───────────────────┐  read/write  ┌──────────┐
│  Device  │               │  server-side attr  │◄────────────►│ Platform │
└──────────┘               └───────────────────┘              └──────────┘
```

Server-side attributes store platform-managed metadata and configuration. **Devices cannot read or modify them** — they are managed exclusively via the ThingsBoard UI or REST API.

**Typical use:** location data, alarm thresholds, operational settings.

```json
{
  "latitude": 40.7128,
  "longitude": -74.0060,
  "maxTemperatureThreshold": 40,
  "alarmEnabled": true
}
```

- _latitude_ / _longitude_ — used in map widgets to show entity location
- _maxTemperatureThreshold_ — referenced by the Rule Engine to trigger alarms
- _alarmEnabled_ — controls whether alarm generation is active

### Shared attributes

```
┌──────────┐   read only   ┌─────────────┐      write      ┌──────────┐
│  Device  │◄──────────────│  shared attr│◄────────────────│ Platform │
└──────────┘               └─────────────┘                 └──────────┘
```

Shared attributes deliver configuration from the platform to a device. The platform writes them; the device can read them but **cannot modify them**.

**Typical use:** remote device configuration — target firmware version, reporting interval, operational thresholds.

Devices receive updates depending on their protocol:
- **Real-time** (e.g., <DocLink product={props.product} path='TODO'>MQTT</DocLink>) — subscribe to attribute updates and receive notifications immediately when a value changes
- **Periodic polling** (e.g., <DocLink product={props.product} path='TODO'>HTTP</DocLink>) — poll the server at intervals for the latest values
- **Downlink** (network integrations) — updates delivered as <DocLink product={props.product} path='TODO'>downlink messages</DocLink>

```json
{
  "targetFirmwareVersion": "v2.4.0",
  "reportingInterval": 60,
  "maxTemperatureThreshold": 45
}
```

- _targetFirmwareVersion_ — device checks this to decide whether an upgrade is needed
- _reportingInterval_ — how often (in seconds) the device should publish telemetry
- _maxTemperatureThreshold_ — limit the device uses locally for its own control logic

### Client-side attributes

```
┌──────────┐     write      ┌──────────────────┐  read only  ┌──────────┐
│  Device  │───────────────►│ client-side attr  │────────────►│ Platform │
└──────────┘                └──────────────────┘             └──────────┘
```

Client-side attributes are reported by the device to the platform. The platform stores and displays them but **cannot modify them** — only the device can write these values.

**Typical use:** reporting active firmware version, current configuration, device state.

```json
{
  "currentFirmwareVersion": "v2.3.1",
  "currentConfiguration": { "mode": "auto", "interval": 30 },
  "deviceState": "ACTIVE"
}
```

- _currentFirmwareVersion_ — used for fleet-wide firmware monitoring
- _currentConfiguration_ — the device's active runtime config (mode, interval, etc.)
- _deviceState_ — current status, e.g., `ACTIVE`, `IDLE`, `ERROR`

## Managing Attributes

### Via UI

<Steps>
1. Open the entity details page (e.g., **Entities → Devices → [device name]**).
2. Navigate to the **Attributes** tab and select the scope: **Server-side**, **Shared**, or **Client-side**.
3. Click **+** to open the **Add attribute** dialog.
4. Enter the **Key**, select the **data type** (String, Boolean, Integer, Double, or JSON), and enter the **Value**.
5. Click **Add**.
</Steps>

<Aside type="note">
Client-side attributes cannot be created or modified from the UI — they are published by the device.
</Aside>

For bulk creation, use <DocLink product={props.product} path='TODO'>bulk provisioning</DocLink> to import multiple entities and their attributes from a CSV file.

### Via REST API

Attributes can be read and written programmatically via the <DocLink product={props.product} path='TODO'>REST API</DocLink>. Authenticate using a JWT token in the `X-Authorization` header. Separate endpoints exist for each scope.

## Performance

Enable `CACHE_ATTRIBUTES_ENABLED` to load each attribute from the database only once and serve subsequent reads from cache.

<Aside type="tip">
If using Redis or Valkey, set `maxmemory-policy` to `allkeys-random` to prevent the cache from exhausting available memory.
</Aside>
