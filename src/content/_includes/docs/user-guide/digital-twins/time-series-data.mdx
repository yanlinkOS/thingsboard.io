**Time-series data** is a sequence of timestamped key-value pairs published by a device or computed by the platform. It represents continuously changing measurements — sensor readings, counters, state changes — where **every data point matters**, not just the latest.

Compared to <DocLink product={props.product} path='TODO'>attributes</DocLink>:

| | Attributes | Time-series |
|--|--|--|
| Storage | Latest value only | Full history |
| Typical data | Config, state | Sensor readings |
| Queryable by time range | No | Yes |
| Aggregatable | No | Yes |

## Data Model

Each time-series entry consists of:
- **Key** — the measurement name (e.g., `temperature`)
- **Value** — string, boolean, integer, double, or JSON
- **Timestamp** — Unix epoch in milliseconds; auto-assigned by the server if not provided by the device

**Simple payload:**
```json
{ "temperature": 26.5, "humidity": 78, "pressure": 1013.2 }
```

**With explicit timestamp:**
```json
{ "ts": 1745000000000, "values": { "temperature": 26.5, "humidity": 78 } }
```

> Use [camelCase](https://en.wikipedia.org/wiki/Camel_case) for key names. This simplifies JavaScript processing in the Rule Engine and dashboards.

## Publishing Telemetry

Devices publish time-series data using the telemetry upload API over multiple protocols:
- **MQTT** — topic `v1/devices/me/telemetry`
- **HTTP** — `POST /api/v1/{accessToken}/telemetry`
- **CoAP**, **LwM2M**, and others via <DocLink product={props.product} path='TODO'>integrations</DocLink>

**Batch upload** — send multiple readings with different timestamps in a single request:
```json
[
  { "ts": 1745000000000, "values": { "temperature": 25.1 } },
  { "ts": 1745000060000, "values": { "temperature": 25.4 } },
  { "ts": 1745000120000, "values": { "temperature": 26.0 } }
]
```

## Storage and Retention

ThingsBoard stores time-series data in PostgreSQL by default. For high-volume deployments, <DocLink product={props.product} path='TODO'>Cassandra</DocLink> and <DocLink product={props.product} path='TODO'>TimescaleDB</DocLink> are supported as alternative storage backends.

A separate **latest values** table is maintained per key, enabling fast access to the most recent reading without scanning the full history.

**Data retention (TTL)** can be configured at the tenant or entity level to automatically delete data older than a defined period, keeping storage usage predictable.

## Aggregation

When querying historical data, ThingsBoard can aggregate raw points into time intervals:

| Function | Result |
|----------|--------|
| NONE | Raw data points |
| AVG | Average value per interval |
| MIN / MAX | Minimum / maximum value per interval |
| SUM | Sum of values per interval |
| COUNT | Number of data points per interval |

**Example:** 1,440 temperature readings over 24 hours can be aggregated into 24 hourly averages — reducing data volume while preserving trends.

Aggregation is controlled by specifying an **interval** and **aggregation function** in dashboard widgets or REST API queries.

## Dashboards

Time-series data is primarily visualized with the <DocLink product={props.product} path='TODO'>Time-series chart</DocLink> widget, and is also supported by:
- **Bar chart** — compare values across time intervals
- **Entity table** — show latest values across multiple entities
- **Gauges** (digital and analog) — display the most recent value

Each widget datasource can be configured with a **time window** and **aggregation function** to control what data is fetched and how it is displayed.

## REST API

Time-series data can be queried and managed programmatically via the <DocLink product={props.product} path='TODO'>REST API</DocLink>. The API supports:
- Fetching **latest values** for a list of keys
- Fetching **historical data** with a time range, interval, aggregation function, and result limit
- **Deleting** data for specific keys and time ranges

Authenticate using a JWT token in the `X-Authorization` header.
