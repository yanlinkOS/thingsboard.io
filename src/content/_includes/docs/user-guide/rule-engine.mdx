The Rule Engine is the data processing backbone of ThingsBoard. Every incoming telemetry reading,
attribute update, device lifecycle event, or RPC call becomes a **Message** that flows through a
**Rule Chain** — a visual pipeline of **Rule Nodes** that filter, enrich, transform, or act on it.

<ImageGallery images={[{ src: '/src/assets/schemas/rule-engine-overview.svg', alt: 'Rule Engine overview', caption: 'A Message (with type, payload, and metadata) flows into the Root Rule Chain, where Filter, Transform, and Action nodes route it to storage, alarms, or other systems.' }]} />

## Message

A Rule Engine Message is an immutable, serializable data structure representing a single event:

| Field | Description |
|-------|-------------|
| **Message ID** | Time-based universally unique identifier |
| **Originator** | The entity that generated the message (Device, Asset, etc.) |
| **Type** | What triggered the message — e.g. `POST_TELEMETRY_REQUEST`, `INACTIVITY_EVENT` |
| **Payload** | JSON body with the event data |
| **Metadata** | Key-value pairs with contextual data (device name, type, custom fields added by nodes) |

For the full list of built-in type strings, originators, metadata fields, and payload shapes, see
<DocLink product={props.product} path="reference/rule-engine/message-types">Rule Engine Message Types</DocLink>.

## Rule Node

A Rule Node processes one incoming message at a time and produces one or more outgoing messages.
Nodes are the building blocks of Rule Chains and fall into seven categories — see
<DocLink product={props.product} path="user-guide/rule-nodes">Rule Nodes</DocLink> for the full reference.

## Rule Chain

A Rule Chain is a directed graph of Rule Nodes connected by **relations**. When a node produces
an outgoing message, the relation type (e.g. `Success`, `Failure`, `True`, `False`, or a custom
name) determines which connected nodes receive it next.

<ImageGallery images={[{ src: '/src/assets/schemas/rule-chain.svg', alt: 'Rule Chain example', caption: 'Message Type Switch routes Post Telemetry messages to a Script Filter; True outputs go to Save Telemetry, False outputs go to Log Other.' }]} />

The **Root Rule Chain** is the default entry point for all incoming messages. There is exactly
one Root Rule Chain per tenant at a time. A tenant administrator can define any number
of additional chains; both the root and additional chains can forward messages to each other
using the **Rule Chain** flow node.

**Device profile routing** — a
<DocLink product={props.product} path="user-guide/device-profiles">Device Profile</DocLink>
can override routing for all its devices with two settings:

| Setting | Effect |
|---------|--------|
| **Default rule chain** | Messages from devices of this profile are routed to the specified chain instead of the Root Rule Chain |
| **Queue** | Messages from devices of this profile are placed in the specified queue instead of the default **Main** queue |

This lets you isolate processing logic and queue priorities per device type without modifying
the root chain.

The Root Rule Chain and additional Rule Chains can be created and managed only by **Tenant Administrators**.

## Message Processing Results

Every message processing attempt ends as **Success**, **Failure**, or **Timeout**:

| Result | When it occurs |
|--------|----------------|
| **Success** | The last node in the processing path completes successfully |
| **Failure** | A node produces a Failure output and no downstream node handles it |
| **Timeout** | Overall processing time exceeds the configured threshold |

A message is only marked as **Failure** if the failure output is **unhandled**. Connecting a node
to the Failure relation (e.g. a log or fallback write) means the failure is handled and the
message can still end as Success:

<ImageGallery images={[{ src: '/src/assets/schemas/message-processing-results.svg', alt: 'Message processing results', caption: 'A Transform failure routed to Save to DB is handled — the overall result is still Success. An unhandled failure produces a Failure result; no reply within the timeout produces a Timeout result.' }]} />

To track success, failure, and timeout counts across all rule chains, use the
<DocLink product={props.product} path="user-guide/rule-engine/monitoring#statistics">statistics dashboard</DocLink>.
To investigate why a specific node failed, enable
<DocLink product={props.product} path="user-guide/rule-engine/monitoring#debug">debug mode</DocLink>
on that node.

## Managing Rule Chains

### Create a Rule Chain

<Steps>
1. Go to **Rule chains** in the left sidebar.
2. Click **+ Add rule chain &#8702; Create new rule chain** in the top-right corner.
3. Enter a name and click **Add**.
4. Click the rule chain row to open the editor and configure nodes.
</Steps>

### Export a Rule Chain

<Steps>
1. Go to **Rule chains** in the left sidebar.
2. Click the **Export rule chain** icon in the desired row.
</Steps>

A JSON file is saved to your computer.

### Import a Rule Chain

<Steps>
1. Go to **Rule chains** in the left sidebar.
2. Click **+ Add rule chain &#8702; Import rule chain** in the top-right corner.
3. Upload the JSON file and click **Import**.
</Steps>

<Aside type="note">
  Imported rule chains are **not** set as Root. If the chain contains **Rule Chain** nodes that
  reference other chains, update those references before saving.
</Aside>

### Set a Rule Chain as Root

<Steps>
1. Go to **Rule chains**.
2. Click the flag icon in the desired row and confirm.
</Steps>

### Create a Nested Rule Chain

<Steps>
1. Go to **Rule chains** in the left sidebar.
2. Open the chain that contains the nodes you want to extract.
3. Select the nodes **and** the links between them — drag a selection rectangle around the
   group, or **Ctrl-click** each node and each link individually. Without the links in the
   selection, **Create nested rule chain** stays disabled.
4. Press **Ctrl+R**, or **right-click** the selection and choose **Create nested rule chain**.
   ThingsBoard creates a new rule chain from the selected nodes and replaces them in the
   current chain with a single <DocLink product={props.product} path="reference/rule-engine/nodes/flow/rule-chain/">Rule Chain</DocLink> node pointing at it.
</Steps>

### Forward Messages to Another Rule Chain

<Steps>
1. Go to **Rule chains** in the left sidebar.
2. Open the rule chain that will reference the target chain.
3. From the **Flow** palette, drag a **Rule Chain** node onto the canvas.
4. In the node's configuration, select the target rule chain.
5. Wire one or more connections from the relevant node output of your existing flow into the
   <DocLink product={props.product} path="reference/rule-engine/nodes/flow/rule-chain/">Rule Chain</DocLink> node, then save.
</Steps>

### Delete a Rule Chain

<Steps>
1. Go to **Rule chains**.
2. Click the **Delete** icon in the desired row — or select multiple rows and use **Delete selected**.
</Steps>

## Configuration

To configure a rule node, double-click it in the Rule Chain editor. The configuration window is
specific to each node type.

Nodes with <DocLink product={props.product} path="user-guide/tbel" target="_blank">TBEL</DocLink>/JS scripts provide a **Test** button. Use it to verify the script with custom
inputs before deploying:

| Panel | Content |
|-------|---------|
| **Top-left** | Message type |
| **Left** | Message payload |
| **Right** | Metadata |
| **Script area** | The TBEL/JS function |
| **Output** | Execution result |

## Monitoring

ThingsBoard provides a **Statistics** dashboard for tracking throughput, failures, and
exceptions across all queues, and **Debug** mode for per-node message tracing. See
<DocLink product={props.product} path="user-guide/rule-engine/monitoring" target="_blank">Rule Engine Monitoring</DocLink>
for the full reference, including the debug event columns, the two debug modes, and the

## Queues

Rule Engine uses message queues to guarantee delivery under load, handle traffic spikes, and
control the order in which messages are processed. See
<DocLink product={props.product} path="user-guide/rule-engine/queues" target="_blank">Queues</DocLink> for
configuration details, submit and retry strategies, and the three built-in queues.

## Custom REST API Calls

ThingsBoard provides an HTTP API to send custom requests directly to the Rule Engine and return
the processing result in the response body — useful for extending the platform API, enriching
calls with device/asset attributes, or powering custom widgets.

Use the `rule-engine-controller` endpoints from the
**REST API reference**.
The entity ID in the request becomes the message originator. If omitted, the calling user entity is used.

## Quick Start: Temperature Validation

A DHT22 sensor reports temperatures between −40 °C and +80 °C. Any reading outside that range
should be discarded and logged instead of saved.

<Steps>
1. Go to **Rule chains** and open the **Root Rule Chain**.
2. Drag a **Script Filter** node onto the canvas. Enter title **Temperature Validation** and
   write the script:
   ```js
   return typeof temperature === 'undefined' || (temperature >= -40 && temperature <= 80);
   ```
   Click **Add**.
3. Delete the **Post Telemetry** relation between **Message Type Switch** and **Save Timeseries**
   (hover the connection arrow and click ✕).
4. Draw a **Post Telemetry** relation from **Message Type Switch** to **Temperature Validation**.
5. Draw a **True** relation from **Temperature Validation** to **Save Timeseries**.
6. Draw a **False** relation from **Temperature Validation** to **Log Other**.
7. Click **Apply changes**.
</Steps>

**Test — out-of-range value (99 °C):**

```bash
curl -X POST -d '{"temperature":99}' \
  http://localhost:8080/api/v1/$ACCESS_TOKEN/telemetry \
  -H "Content-Type:application/json"
```

The value does **not** appear in Latest Telemetry — filtered and logged.

**Test — valid value (24 °C):**

```bash
curl -X POST -d '{"temperature":24}' \
  http://localhost:8080/api/v1/$ACCESS_TOKEN/telemetry \
  -H "Content-Type:application/json"
```

The value appears in Latest Telemetry — saved successfully.
