ThingsBoard Platform Integrations connect devices that use vendor-specific protocols — LoRaWAN network servers, cloud IoT platforms, industrial OPC-UA servers — to ThingsBoard without changing device firmware.
Two primary deployment scenarios:

- **Connect NB-IoT, LoRaWAN, SigFox, and other devices** with proprietary payloads directly to ThingsBoard.
- **Stream data from existing IoT platforms** (AWS IoT, Azure IoT Hub, etc.) to enable real-time dashboards and processing.

In both cases a server-side component sits between ThingsBoard and the device, exposing its own API.
The integration bridges that API to ThingsBoard core features: telemetry collection, attributes, and RPC calls.

## How integrations work

ThingsBoard supports many integration protocols — HTTP, MQTT, OPC-UA, CoAP, TCP, UDP — and specific connectors for LoRaWAN network servers (ChirpStack, The Things Stack, Loriot), Sigfox, AWS IoT, Azure Event Hub, and more.

<DocImage src="/src/assets/images/user-guide/integrations/overview/create-integration-1.png" alt="Create integration" />

```plaintext
  External Platform / Device
          │
          ▼
  ┌──────────────────────────────────┐
  │       ThingsBoard Integration    │
  │  (validates payload & security)  │
  └──────────────┬───────────────────┘
                 │
     ┌───────────┴───────────┐
     ▼                       ▼
  Uplink Converter      Downlink Converter
  (device → TB format)  (TB → device format)
     │                       ▲
     ▼                       │
  ┌──────────────────────────────────┐
  │     ThingsBoard Rule Engine      │
  │  (telemetry, attributes, RPC)   │
  └──────────────────────────────────┘
```

**Uplink flow:** An incoming message is validated, then the assigned **Uplink Data Converter** extracts meaningful fields (telemetry, attributes, device name) and passes them to the Rule Engine.

**Downlink flow:** The Rule Engine pushes a message (e.g. shared attribute update, RPC call) through the assigned **Downlink Data Converter**, which transforms it into the format expected by the external platform.

Common downlink use cases:

- Changing data upload frequency via a shared attribute update
- Triggering a firmware update
- Sending an RPC command to change device state

## Uplink data converter

The uplink converter parses incoming device payloads and converts them into ThingsBoard's internal format.

### Generic uplink converter

A generic converter works with any integration type. To create one:

<Steps>
1. Go to **Integration center** → **Data converters**, click **+**, then **Create new converter**.
2. Ensure the type selector is set to **Uplink**. Select the integration type (MQTT, OPC-UA, etc.) or leave it as "All".
3. Enter a name and optionally enable **Debug mode**.
4. In **Main decoding configuration**, write or paste your decoder script.
5. Configure **Advanced decoding parameters**: use the "Update only keys list" to skip unchanged values.
6. Click **Add**.
</Steps>

<Aside type="tip">
We recommend using <DocLink product={props.product} path="user-guide/tbel">TBEL</DocLink> (ThingsBoard Expression Language) for converter scripts — it executes much more efficiently than JavaScript inside ThingsBoard.
</Aside>

### Typed uplink converter

Starting from **ThingsBoard 4.0**, certain integrations support a simplified typed uplink converter where you map message fields to attributes or telemetry without writing a decoder function manually.

Supported integrations: <DocLink product={props.product} path="user-guide/integrations/chirpstack">ChirpStack</DocLink>, <DocLink product={props.product} path="TODO">Loriot</DocLink>, <DocLink product={props.product} path="TODO">The Things Stack Community</DocLink>, <DocLink product={props.product} path="TODO">The Things Stack Industries</DocLink>, ThingPark, and ThingPark Enterprise.

<Aside type="note">Converters created before ThingsBoard 4.0 continue to work without changes.</Aside>

To create a typed uplink converter:

<Steps>
1. In the converter creation dialog, select the **Integration type** (e.g. Loriot).
2. Enter a converter name.
3. In **Main decoding configuration**, select the entity type (**Device** or **Asset**) and specify the entity name pattern. Use `$` patterns (e.g. `$eui`) to dynamically populate values from the incoming message.
4. Optionally set **Device profile**, **Device label**, **Customer name**, and **Device group name** (all support `$` patterns).
5. In **Attributes** and **Telemetry** sections, specify which keys go where. Missing keys are ignored.
6. Configure the "Update only keys list" if needed, then click **Add**.
</Steps>

You can validate your decoder logic before saving. Click **Test payload decoder** to open the test window.

**Payload** — the encoded input data from the device. Each integration type has a pre-filled sample. Supported formats:
- **JSON**
- **TEXT**
- **Binary (Base64)** — automatically decoded to a byte array; no manual decoding needed in the script.

Access the payload in your script using the `payload` keyword.

**Metadata** — a key-value map of integration-specific fields, accessible via the `metadata` keyword. You can define custom metadata parameters in each integration's settings (e.g. `deviceType`).

**Converter output** — the JSON result from your decoder. The only mandatory fields are `deviceName` and `deviceType` (or `assetName` and `assetType` for assets). Optional fields: `customerName`, `groupName` (auto-create customer/group and assign the entity).

Example output:

```json maxLines=15 collapsible
{
  "entityType": "DEVICE",
  "name": "Device 1000000000000001",
  "profile": "default",
  "telemetry": [
    {
      "ts": 1684478801936,
      "values": {
        "battery": 95,
        "temperature": 36.6,
        "saturation": 99
      }
    },
    {
      "rssi": -21,
      "data": "01755e030001040001",
      "battery": 94,
      "snr": 10

  ],

    "serialNumber": "SN-12345678",
    "fPort": 85,
    "dr": "SF9 BW125 4/5",
    "frequency": 867500000,
    "eui": 1000000000000001
  }
}
```

## Downlink data converter

The downlink converter transforms a Rule Engine message into the format expected by the external integration.

<Aside type="note">A downlink converter is optional. You only need one if the integration sends outgoing messages (RPCs, attribute updates, commands). Receive-only integrations can skip it.</Aside>

To create a downlink converter:

<Steps>
1. Go to **Integration center** → **Data converters**, click **+**, then **Create new converter**.
2. Switch the type selector to **Downlink**. Select the integration type or leave "All".
3. Enter a name and optionally enable **Debug mode**.
4. In **Main encoding configuration**, write your encoder script.
5. Click **Add**.
</Steps>

<DocImage src="/src/assets/images/user-guide/integrations/overview/add-downlink-converter.png" alt="Add downlink converter" />

The encoder function signature:

```js
function encoder(msg, metadata, msgType, integrationMetadata);
```

| Parameter | Description |
|---|---|
| `msg` | JSON message payload from the Rule Engine |
| `metadata` | Key-value pairs with additional message data |
| `msgType` | Rule Engine message type (e.g. `ATTRIBUTES_UPDATED`, `POST_TELEMETRY_REQUEST`) |
| `integrationMetadata` | Key-value map with integration-specific fields configured in integration details |

### Converter output

The encoder must return a JSON object:

```json
{
  "contentType": "JSON",
  "data": "{\"tempFreq\":60,\"firmwareVersion\":\"1.2.3\"}",
  "metadata": {
    "topic": "temp-sensor/sensorA/upload"
  }
}
```

| Field | Description |
|---|---|
| `contentType` | `JSON`, `TEXT`, or `BINARY` (Base64 string) |
| `data` | Encoded data string matching the content type |
| `metadata` | Key-value pairs — e.g. MQTT topic, HTTP headers |

### Synchronous vs asynchronous downlinks

Most integrations (MQTT, AWS IoT, etc.) push downlinks asynchronously — each Rule Engine message is sent to the external broker immediately.

Some integrations (SigFox, generic HTTP) can only push downlinks **synchronously** in reply to an uplink request. In this case, the last downlink message is queued until the next uplink arrives for that device.

### Example

Suppose temperature and humidity upload frequency attributes are updated via the REST API, and you want to push the update to an external MQTT broker. You also want to include the `firmwareVersion` attribute. The topic should contain the device name.

<Tabs>
<TabItem label="TBEL (Recommended)">
```js maxLines=20 collapsible
// Encode downlink data from incoming Rule Engine message
// msg - JSON message payload
// msgType - message type, e.g. 'ATTRIBUTES_UPDATED'
// metadata - key-value pairs with additional data
// integrationMetadata - integration-specific fields

var data = {};

data.tempFreq = msg.temperatureUploadFrequency;
data.humFreq = msg.humidityUploadFrequency;
data.firmwareVersion = metadata['ss_firmwareVersion'];

var result = {
    contentType: "JSON",
    data: JSON.stringify(data),
    metadata: {
        topic: metadata['deviceType'] + '/' + metadata['deviceName'] + '/upload'
    }
};

return result;
```
</TabItem>

## Converters library

The Converters Library is a built-in collection of ready-to-use uplink decoder functions for over 100 devices, supporting six popular LoRaWAN network servers. It significantly simplifies integration setup.

Currently supported by: <DocLink product={props.product} path="user-guide/integrations/chirpstack">ChirpStack</DocLink>, <DocLink product={props.product} path="TODO">Loriot</DocLink>, <DocLink product={props.product} path="TODO">The Things Stack Community</DocLink>, <DocLink product={props.product} path="TODO">The Things Stack Industries</DocLink>, ThingPark, and ThingPark Enterprise.

<Steps>
1. Go to **Integration center** → **Integrations**, click **+** to create a new integration. Select a supported integration type and click **Next**.
2. In the **Uplink data converter** tab, switch to the **Library** option. Choose the **device vendor**, then the specific **sensor model**. The decoder function loads automatically.
3. Configure advanced decoding options if needed, then click **Next** to continue setup.
</Steps>

## Debug mode

Debug mode is useful for development and troubleshooting. However, it increases disk usage since all debug data is stored in the database.

Starting from **ThingsBoard 3.9**, the platform stores the full set of debug events only during the first **15 minutes** after creation. After that, only failure events are retained. These settings can be combined or disabled entirely.

## Deployment options

### Embedded integrations

Embedded integrations run inside the main ThingsBoard server process (monolith deployment).

| Pros | Cons |
|---|---|
| Simple setup — just a few clicks in the UI | Consumes resources from the main ThingsBoard process |
| Minimal message delivery latency | Low isolation from core platform |
| | Cannot reach local MQTT brokers / OPC-UA servers when ThingsBoard runs in the cloud |

<DocImage src="/src/assets/images/user-guide/integrations/embeded-integrations-overview.svg" alt="Embedded integrations architecture" />

### Remote integrations

Remote integrations run as a separate process, typically deployed close to the data source (e.g. a local MQTT broker or OPC-UA server that has no external IP). The remote integration pulls data and streams it to the cloud-hosted ThingsBoard instance when internet is available.

| Pros | Cons |
|---|---|
| Connects to servers in a local network | Requires installing a separate package |
| Isolates integration from the main ThingsBoard process | |
| Buffers data locally when internet is unavailable | |

<DocImage src="/src/assets/images/user-guide/integrations/remote-integrations-overview.svg" alt="Remote integrations architecture" />

## Integrations vs IoT Gateway

| | Platform Integrations | <DocLink product={props.product} path="TODO">IoT Gateway</DocLink> |
|---|---|---|
| **Designed for** | Server-to-server communication | Local network deployments |
| **Scale** | High throughput, cluster-ready | Up to ~1,000 devices |
| **Custom decoders** | Modify converter scripts in real time via UI | Requires gateway recompilation and restart |

Both systems serve different use cases and complement each other.
