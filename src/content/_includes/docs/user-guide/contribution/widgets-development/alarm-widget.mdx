Build a **custom alarm table** that displays active alarms with severity-based colors and lets users acknowledge or clear alarms directly from the widget.

## What you'll build

An alarm widget with:
- A table listing alarms with columns for time, type, severity, and status
- Severity-based row colors (Critical = red, Major = orange, Minor = yellow, Warning = blue)
- Acknowledge and Clear buttons on each row
- Auto-refresh when new alarms arrive

## Alarm widget basics

**Alarm** widgets subscribe to alarm data instead of telemetry. The platform delivers alarms matching the configured filter (severity, status, type) through `self.ctx.defaultSubscription.alarms`.

Each alarm object contains:

| Property | Type | Description |
|----------|------|-------------|
| `id.id` | string | Alarm ID |
| `type` | string | Alarm type (e.g., "High Temperature") |
| `severity` | string | `CRITICAL`, `MAJOR`, `MINOR`, `WARNING`, `INDETERMINATE` |
| `status` | string | `ACTIVE_UNACK`, `ACTIVE_ACK`, `CLEARED_UNACK`, `CLEARED_ACK` |
| `createdTime` | number | Timestamp when the alarm was created (ms) |
| `ackTs` | number | Timestamp when acknowledged (0 if not) |
| `clearTs` | number | Timestamp when cleared (0 if not) |
| `originator` | object | Entity that raised the alarm (`{ entityType, id }`) |
| `originatorName` | string | Display name of the originator entity |

4. In the **JavaScript** tab, paste:

   ```javascript
   let alarmService;

   self.onInit = function() {
     let $injector = self.ctx.$scope.$injector;
     alarmService = $injector.get(
       self.ctx.servicesMap.get('alarmService'));

     self.ctx.$scope.alarms = [];

     self.ctx.$scope.formatTime = function(ts) {
       return moment(ts).fromNow();
     };

     self.ctx.$scope.ackAlarm = function(alarm) {
       alarmService.ackAlarm(alarm.id.id).subscribe(
         function() {
           alarm.ackTs = Date.now();
           self.ctx.detectChanges();
         }
       );
     };

     self.ctx.$scope.clearAlarm = function(alarm) {
       alarmService.clearAlarm(alarm.id.id).subscribe(
         function() {
           alarm.clearTs = Date.now();
           self.ctx.detectChanges();
         }
       );
     };
   };

   self.onDataUpdated = function() {
     let subscription = self.ctx.defaultSubscription;
     if (subscription && subscription.alarms) {
       self.ctx.$scope.alarms = subscription.alarms.data;
     }
     self.ctx.detectChanges();
   };

   self.typeParameters = function() {
     return {
       hasDataPageLink: true,
       singleEntity: false,
       warnOnPageDataOverflow: false
     };
   };

   self.actionSources = function() {
     return {
       'rowClick': {
         name: 'widget-action.row-click',
         multiple: false
       }
     };
   };
   ```

   **Key concepts:**

   **Alarm data access** — in `onDataUpdated()`, alarms are available at `self.ctx.defaultSubscription.alarms.data`. This is an array of alarm objects. The platform handles filtering by severity, status, and type based on the widget's alarm configuration.

   **Alarm service** — inject `alarmService` to perform alarm operations:
   - `alarmService.ackAlarm(alarmId)` — acknowledge an alarm
   - `alarmService.clearAlarm(alarmId)` — clear an alarm
   Both return Observables — subscribe to get notified when the operation completes.

   **Type parameters** — `hasDataPageLink: true` enables server-side pagination for large alarm lists. `singleEntity: false` allows alarms from multiple entities.

## Alarm status reference

| Status | Acknowledged | Cleared | Description |
|--------|-------------|---------|-------------|
| `ACTIVE_UNACK` | No | No | New alarm, not yet seen |
| `ACTIVE_ACK` | Yes | No | Acknowledged but still active |
| `CLEARED_UNACK` | No | Yes | Condition resolved, not acknowledged |
| `CLEARED_ACK` | Yes | Yes | Fully resolved |
