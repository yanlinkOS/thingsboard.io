ThingsBoard <DocLink product={props.product} path="user-guide/dashboards/" target="_blank">dashboards</DocLink> provide interactive visualization for monitoring and managing IoT data. Key capabilities include data visualization through charts, graphs, and tables; device management; interactive control elements; customizable layouts; and role-based access controls.

A ThingsBoard dashboard is built from five core building blocks. Understanding how they connect will help you design effective dashboards.

The following diagram shows a typical drill-down dashboard. The **Main** state lists all devices in an Entities Table. When a user clicks a row, an **action** fires and navigates to the **Device Details** state, where a chart shows data for the selected device. Each state has its own **layout** and **time window**, and each widget gets data through an **alias**.

## States

<DocLink product={props.product} path="user-guide/dashboards/#states" target="_blank">States</DocLink> are named views within a single dashboard that form a navigation hierarchy. Every dashboard has at least one state — the **root state**. You can add additional states to create detail pages, configuration panels, or any other sub-views.

**When to use:** Drill-down from a device/asset list to individual entity details without leaving the dashboard. For example, a root state lists all thermostats, and a detail state shows temperature charts for the selected thermostat.

<ImageGallery images={[{ src: '/src/assets/schemas/key-concepts-states.svg', alt: 'Dashboard states navigation', caption: 'A row click in the root state navigates to the detail state for the selected device; a back button returns to the root state.' }]} />

## Entity Aliases

<DocLink product={props.product} path="user-guide/aliases/" target="_blank">Entity Aliases</DocLink> is a named reference that tells a widget where to get data. Aliases can be **static** (pointing to a fixed entity) or **dynamic** (resolved at runtime from dashboard state, relations, device type, etc.).

Not every widget requires an alias — static widgets like HTML cards or navigation buttons may have no data source. But any widget that displays entity data (telemetry, attributes, alarms) needs at least one.

**When to use:** Use static aliases for dashboards that always show the same entity. Use dynamic aliases — especially <DocLink product={props.product} path="user-guide/aliases/#entity-from-dashboard-state" target="_blank">Entity from dashboard state</DocLink> — for drill-down scenarios where the alias resolves to whichever entity the user selected in the parent state.

## Layouts

Layouts control how widgets are arranged on the dashboard grid. See <DocLink product={props.product} path="user-guide/layouts/" target="_blank">Layouts</DocLink> for layout settings, breakpoints, and configuration details.

Each <DocLink product={props.product} path="user-guide/data-visualization/#states">state</DocLink> has its own layout. Three types are available:

- <DocLink product={props.product} path="user-guide/layouts/#default-layout" bold={true} target="_blank">Default</DocLink> — widgets snap to a column-based grid. Supports drag-and-drop positioning. If <DocLink product={props.product} path="user-guide/layouts/#auto-fill-layout-height" target="_blank">auto-fill layout height</DocLink> is enabled, widgets redistribute vertically to fill the screen.
- <DocLink product={props.product} path="user-guide/scada" bold={true} target="_blank">SCADA</DocLink> — designed for industrial HMI screens. No margin between widgets, so pipes and SCADA symbols always connect properly. Widgets are placed with a transparent background and no shadow by default, blending into the SCADA canvas.
- <DocLink product={props.product} path="user-guide/layouts/#divider" bold={true} target="_blank">Divider</DocLink> — splits the dashboard into left and right panels, useful for master-detail views (e.g., entity list on the left, entity details on the right).

<ImageGallery images={[{ src: '/src/assets/schemas/layouts.svg', alt: 'Dashboard layout types', caption: 'Three layout types: Default (column grid), SCADA (seamless canvas with no margins), and Divider (split left/right panels).' }]} />

Each layout also supports <DocLink product={props.product} path="user-guide/layouts/#breakpoints" target="_blank">breakpoints</DocLink> — screen-width thresholds that let you define different widget arrangements for desktop, tablet, and mobile. This means a single state can adapt its widget set and positioning for different screen sizes.

## Actions

<DocLink product={props.product} path="user-guide/actions/" target="_blank">Actions</DocLink> are event handlers that respond to user interactions — a row click, a header button press, a map marker tap — and trigger navigation, state updates, URL opens, or custom JavaScript logic.

**When to use:** Wire up drill-down navigation between states, link widgets to external pages, or trigger custom functions (delete entity, open dialog, etc.).

## Time Window

The <DocLink product={props.product} path="user-guide/time-window/" target="_blank">time window</DocLink> controls the time range and aggregation applied to all time-series data on a dashboard.

- **Real-time** — a sliding window that continuously updates (e.g., *"last 1 hour"*). Use for live monitoring dashboards.
- **History** — a fixed time range loaded once (e.g., *"March 1 – March 7"*). Use for analysis and reporting.

<ImageGallery images={[{ src: '/src/assets/schemas/timewindows.svg', alt: 'Time window types', caption: 'Real-time: a sliding window that moves with time. History: a fixed date range loaded once.' }]} />

By default, the **dashboard-level time window** applies to all widgets. Individual time-series widgets can override it using their own time window settings.
