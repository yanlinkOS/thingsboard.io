ThingsBoard provides multiple layers of security for platform access, user authentication, and data protection. This page summarizes the available mechanisms and links to detailed configuration guides.

## User Authentication

ThingsBoard authenticates users via username and password by default. Administrators can enhance sign-in security with additional methods:

| Method | Description |
|---|---|
| **Username and password** | Default sign-in method.{![Products.PAAS, Products.PAAS_EU].includes(props.product) && <> Configure password complexity, expiration, and account lockout rules in <DocLink product={props.product} path="user-guide/security" target="_blank">security settings</DocLink>.</>} |
| **Two-factor authentication** | Adds a second verification step after entering credentials. Supports TOTP authenticator apps, email, SMS, and backup codes. See <DocLink product={props.product} path="user-guide/security/two-factor-authentication" target="_blank">Two-factor authentication</DocLink>. |
| **OAuth 2.0 / SSO** | Allows sign-in via external identity providers like Google, Auth0, Keycloak, Okta, and Azure AD. See <DocLink product={props.product} path="user-guide/security/oauth-2-support" target="_blank">OAuth 2.0 support</DocLink>. |
| **Self-registration** | Lets tenants expose a sign-up page so customers can create their own accounts. See <DocLink product={props.product} path="user-guide/security/self-registration" target="_blank">Self-registration</DocLink>. |

## Programmatic Access

Applications and scripts authenticate to the ThingsBoard REST API using one of two methods:

| Method | Description |
|---|---|
| **JWT tokens** | Obtained by calling the login endpoint with username and password. Tokens expire periodically and must be refreshed.{![Products.PAAS, Products.PAAS_EU].includes(props.product) && <> Configure token lifetime in <DocLink product={props.product} path="user-guide/security" target="_blank">security settings</DocLink>.</>} See the <DocLink product={props.product} path="reference/rest-api" target="_blank">REST API reference</DocLink> for details. |
| **API keys** | Long-lived credentials that bypass the login flow. Ideal for third-party integrations, scripts, and automation. See <DocLink product={props.product} path="user-guide/security/api-keys" target="_blank">API keys</DocLink>. |

### Device Transport Security

Each device connectivity protocol supports encrypted transport and one or more authentication methods:

| Protocol | Encryption | Authentication methods | Reference |
|---|---|---|---|
| **MQTT** | TLS (port 8883) | Access Token, MQTT Basic credentials, X.509 certificate | <DocLink product={props.product} path="reference/mqtt-api/getting-connected" target="_blank">MQTT getting connected</DocLink> |
| **HTTP** | TLS / HTTPS (port 443) | Access Token | <DocLink product={props.product} path="reference/http-api/getting-connected" target="_blank">HTTP getting connected</DocLink> |
| **CoAP** | DTLS (port 5684) | Access Token, X.509 certificate | <DocLink product={props.product} path="reference/coap-api/getting-connected" target="_blank">CoAP getting connected</DocLink> |
| **LwM2M** | DTLS (ports 5686, 5688) | Pre-Shared Key (PSK), Raw Public Key (RPK), X.509 certificate | <DocLink product={props.product} path="reference/lwm2m-api/getting-started" target="_blank">LwM2M getting started</DocLink> |
| **SNMP** | USM authentication + encryption (v3) | Community string (v1/v2c), USM username with auth and privacy passphrases (v3) | <DocLink product={props.product} path="reference/snmp-api/getting-connected" target="_blank">SNMP getting connected</DocLink> |

All protocols support one-way TLS (client verifies the server). MQTT, CoAP, and LwM2M also support mutual TLS, where both sides authenticate using X.509 certificates. For fleet provisioning, you can use certificate chains so that devices sharing a common CA are automatically trusted.
