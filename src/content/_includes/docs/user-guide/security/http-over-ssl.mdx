ThingsBoard supports running HTTP servers with SSL/TLS encryption for the Web UI and REST API. Enabling HTTPS protects traffic between clients and the server.

<Aside type="tip">
  The recommended approach for production deployments is to use HAProxy for SSL/TLS termination. HAProxy sits in front of ThingsBoard and handles certificate management while ThingsBoard continues to operate over plain HTTP internally. Direct SSL configuration on ThingsBoard is an alternative for environments where a reverse proxy is not used.
</Aside>

---

## PEM certificate configuration

Direct SSL support using PEM certificate files is available since ThingsBoard version 3.3.2.

Configure SSL by setting the following environment variables in `thingsboard.conf`, Docker Compose, or Kubernetes manifests:

```bash
export SSL_ENABLED=true
export SSL_CREDENTIALS_TYPE=PEM
export SSL_PEM_CERT=server.pem
export SSL_PEM_KEY=server_key.pem
export SSL_PEM_KEY_PASSWORD=secret
```

| Variable | Description |
|---|---|
| `SSL_ENABLED` | Enables or disables SSL support. Set to `true` to activate HTTPS. |
| `SSL_CREDENTIALS_TYPE` | Credential format: `PEM` for certificate files, or `KEYSTORE` for a Java keystore. |
| `SSL_PEM_CERT` | Path to the server certificate file. The file may also include the private key. |
| `SSL_PEM_KEY` | Path to the private key file. Optional if the private key is already included in the certificate file. |
| `SSL_PEM_KEY_PASSWORD` | Password for an encrypted private key. Optional if the key is not password-protected. |

## Certificate file placement

Certificate files must be accessible to the ThingsBoard process. Place them in the appropriate directory for your deployment platform:

<Tabs>
<TabItem label="Linux">
Place the certificate files in `/etc/thingsboard/conf/` with the same file permissions as `thingsboard.conf`. Use relative paths in the configuration variables:

```bash
export SSL_PEM_CERT=server.pem
export SSL_PEM_KEY=server_key.pem
```
</TabItem>
<TabItem label="Docker Compose">
Mount the directory containing your certificate files to `/config` inside the container. Use full paths in the configuration variables:

```bash
export SSL_PEM_CERT=/config/server.pem
export SSL_PEM_KEY=/config/server_key.pem
```
</TabItem>
<TabItem label="Kubernetes">
Mount the directory containing your certificate files to `/https-config` inside the container. Use full paths in the configuration variables:

```bash
export SSL_PEM_CERT=/https-config/server.pem
export SSL_PEM_KEY=/https-config/server_key.pem
```
</TabItem>
<TabItem label="Windows">
Place the certificate files in `C:\Program Files (x86)\thingsboard\conf\`. Use relative paths in the configuration variables:

```bash
export SSL_PEM_CERT=server.pem
export SSL_PEM_KEY=server_key.pem
```
</TabItem>
</Tabs>

## Generating certificates

You can use OpenSSL to generate certificates for testing or internal environments. For production deployments, use certificates issued by a trusted Certificate Authority (CA).

<Aside type="caution">
  Self-signed certificates are not validated by any third party. Browsers and clients will show a security warning unless the certificate is manually imported and trusted. Use certificates from a trusted CA for production environments.
</Aside>

### Self-signed certificate

Use the following commands to generate a self-signed certificate valid for 365 days:

<Steps>
1. Generate the server private key:

   ```bash
   openssl ecparam -out server_key.pem -name secp256r1 -genkey
   ```

2. Generate the self-signed certificate:

   ```bash
   openssl req -new -key server_key.pem -x509 -nodes -days 365 -out server.pem
   ```

   To skip the interactive prompts and set the Common Name automatically, add `-subj '/CN=localhost'`:

   ```bash
   openssl req -new -key server_key.pem -x509 -nodes -days 365 -out server.pem -subj '/CN=localhost'
   ```
</Steps>

---

### CA-signed certificate

A CA-signed certificate provides better trust control for internal deployments. First create a local Certificate Authority, then use it to sign the server certificate.

<Steps>
1. Generate the CA private key:

   ```bash
   openssl ecparam -out ca_key.pem -name secp256r1 -genkey
   ```

2. Create the self-signed CA certificate (valid 365 days):

   ```bash
   openssl req -new -x509 -key ca_key.pem -days 365 -out ca.pem
   ```

3. Generate the server private key:

   ```bash
   openssl ecparam -out server_key.pem -name secp256r1 -genkey
   ```

4. Create a Certificate Signing Request (CSR) for the server:

   ```bash
   openssl req -new -key server_key.pem -out server.csr
   ```

5. Sign the server certificate with your CA:

   ```bash
   openssl x509 -req -in server.csr -CA ca.pem -CAkey ca_key.pem -CAcreateserial -out server.pem -days 365
   ```

   The `-CAcreateserial` flag automatically creates a `ca.srl` file to track certificate serial numbers.
</Steps>

After completing these steps, configure `SSL_PEM_CERT` and `SSL_PEM_KEY` to point to `server.pem` and `server_key.pem` respectively. Import `ca.pem` into the trust store of any client that needs to verify the server certificate.
