ThingsBoard exposes an HTTP/HTTPS API for devices to publish telemetry, sync attributes, and exchange RPC commands. Any HTTP client — `curl`, an SDK library, or a language's built-in HTTP stack — can connect without a specialized broker library.

## Connection Parameters

| Parameter | Value |
|-----------|-------|
| Host | Your ThingsBoard hostname or IP |
| Port | `80` (HTTP) · `443` (HTTPS) |
| Protocol | HTTP/1.1 |
| Authentication | Access token in the URL path |

## Authentication

HTTP uses **access token** credentials. Include the token as a path segment in every request URL:

<HostCode product={props.product} lang="" code={`{HTTP_URL_PREFIX}/api/v1/$ACCESS_TOKEN/{resource}`} />

Copy the access token from **Entities → Devices → [device] → Copy Access Token**. See <DocLink product={props.product} path='user-guide/devices'>Devices</DocLink> for how to manage device credentials.

<Aside type="note">
  HTTP transport supports **access token** credentials only. MQTT Basic and X.509 certificate credentials cannot be used with the HTTP API.
</Aside>

Where **\{resource\}** defines the endpoint type for your request. The available resources are:
- <DocLink product={props.product} path='reference/http-api/telemetry/'>telemetry</DocLink> — for sending telemetry data;
- <DocLink product={props.product} path='reference/http-api/attributes/'>attributes</DocLink> — for publishing or requesting client/shared attributes;
- <DocLink product={props.product} path='reference/http-api/rpc/'>rpc</DocLink> — for sending or replying to RPC commands;
- <DocLink product={props.product} path='reference/http-api/claiming/'>claiming</DocLink> — for initiating the device claiming process.

## HTTPS (TLS)

Most deployments terminate TLS at a load balancer or reverse proxy (HAProxy, nginx, AWS ALB). The ThingsBoard platform receives plain HTTP traffic internally.

To enable HTTPS directly on the ThingsBoard server, set these environment variables:

```bash
SSL_ENABLED=true
SSL_CREDENTIALS_TYPE=PEM
SSL_PEM_CERT=server.pem
SSL_PEM_KEY=server_key.pem
SSL_PEM_KEY_PASSWORD=secret
```

When connecting over HTTPS with a CA-signed certificate, clients need no special configuration — just use `https://` as the URL scheme. For self-signed certificates, pass `--cacert ca.pem` to `curl`.

## Response Codes

| Code | Meaning |
|------|---------|
| `200 OK` | Client's request has been successfully |
| `400 Bad Request` | Invalid URL, request parameters, or body |
| `401 Unauthorized` | Invalid or missing access token |
| `404 Not Found` | Requested resource does not exist |
| `405 Method Not Allowed` | URL doesn't support this request method |

## Command-Line Tool

Examples throughout this section use **curl**, which is available by default on Linux, macOS, and Windows 10+. Verify with `curl --version`.
