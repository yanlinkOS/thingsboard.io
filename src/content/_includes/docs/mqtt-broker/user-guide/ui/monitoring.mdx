## Charts

At the top of the **Home** page, six charts display broker activity over the last 10 minutes:

- **Incoming messages** — number of messages published by MQTT clients to the broker.
- **Outgoing messages** — count of messages delivered by the broker to subscribing clients.
- **Dropped messages** — messages that lack subscribers or could not be delivered due to network issues, disconnections, or resource limitations.
- **Sessions** — number of current MQTT sessions.
- **Subscriptions** — number of current subscriptions.
- **Network traffic** — size of data processed (sent or received) by MQTT clients.

On the **Monitoring** page, you can zoom in, set custom date ranges, or open charts in full-screen mode.

## Config

The Config card shows commonly used configuration parameters:

- **Basic Authentication** — enable or disable the authentication method with the switch button.
- **X.509 Certificate Chain Authentication**.
- **JWT Authentication**.
- **SCRAM Authentication**.
- **TCP Port** — default `1883` (`LISTENER_TCP_BIND_PORT`).
- **TLS Port** — default `8883`, disabled by default (`LISTENER_SSL_BIND_PORT`).
- **WS Port** — default `8084` (`LISTENER_WS_BIND_PORT`).
- **WSS Port** — default `8085`, disabled by default (`LISTENER_WSS_BIND_PORT`).
- **TCP Listener** — enabled by default (`LISTENER_TCP_ENABLED`).
- **TLS Listener** — disabled by default; set `LISTENER_SSL_ENABLED=true` to enable.
- **WS Listener** — enabled by default (`LISTENER_WS_ENABLED`).
- **WSS Listener** — disabled by default (`LISTENER_WSS_ENABLED`).
- **TCP / TLS / WS / WSS Listener Max Payload Size** — maximum payload size in bytes for each listener (default `65536`; controlled by `TCP_NETTY_MAX_PAYLOAD_SIZE`, `SSL_NETTY_MAX_PAYLOAD_SIZE`, `WS_NETTY_MAX_PAYLOAD_SIZE`, `WSS_NETTY_MAX_PAYLOAD_SIZE`).

## Kafka Consumer Groups

The Kafka Consumer Groups card displays the following information:

- **ID** — Consumer Group ID.
- **State** — can be `STABLE`, `PREPARING_REBALANCE`, `COMPLETING_REBALANCE`, `EMPTY`, `DEAD`, or `UNKNOWN`.
- **Members** — number of consumers in the group.
- **Lag** — sum of all consumer lags within the group (delta between last committed offset and producer's end offset).

## Resource usage statistics

TBMQ provides runtime resource usage statistics for each service instance, helping monitor system behavior and support debugging or optimization.

System metrics are collected using the OSHI Java library, which retrieves hardware and OS statistics directly from the host environment. Metrics are updated every **60 seconds** by default, controlled by:

```yaml
# Persist frequency of system info (CPU, memory usage, etc.) in seconds
persist-frequency: "${STATS_SYSTEM_INFO_PERSIST_FREQUENCY_SEC:60}"
```

The Resource Usage table includes:

- **Last update time** — timestamp of the most recent metrics update.
- **Service ID** — identifier of the service instance.
- **Service type** — type of service (e.g., TBMQ, Integration Executor).
- **CPU** — CPU load percentage (hover to see available CPU cores).
- **RAM** — physical memory usage percentage (hover to see total memory in GB).
- **Disk** — disk space usage percentage (hover to see total disk capacity in GB).
- **Status**:
  - `Active` — reported less than 1 hour ago.
  - `Inactive` — reported between 1 hour and 1 week ago.
  - `Outdated` — reported more than 1 week ago.

TBMQ tracks all registered services in Redis under the key `tbmq:service:registry`. Services are added automatically on first launch and are not removed automatically — you can manually delete `Outdated` services using the **Delete** button in the UI or via REST API.
