## Define environment variables

Define environment variables used throughout this guide. Execute the following command on Linux:

```bash
export AKS_RESOURCE_GROUP=TBMQResources
export AKS_LOCATION=eastus
export AKS_GATEWAY=tbmq-gateway
export TB_CLUSTER_NAME=tbmq-cluster
export TB_DATABASE_NAME=tbmq-db
export TB_REDIS_NAME=tbmq-redis
echo "Variables ready to create resource group $AKS_RESOURCE_GROUP in location $AKS_LOCATION
and cluster $TB_CLUSTER_NAME with database $TB_DATABASE_NAME"
```

Where:
- `TBMQResources` — a logical group in which Azure resources are deployed and managed (**AKS_RESOURCE_GROUP**).
- `eastus` — the region for the resource group (**AKS_LOCATION**). List all regions with `az account list-locations`.
- `tbmq-gateway` — the name of the Azure Application Gateway.
- `tbmq-cluster` — the cluster name (**TB_CLUSTER_NAME**).
- `tbmq-db` — the database server name (**TB_DATABASE_NAME**).

## Configure and create AKS cluster

Create the Azure Resource Group:

```bash
az group create --name $AKS_RESOURCE_GROUP --location $AKS_LOCATION
```

For more details on `az group`, see the <a href="https://learn.microsoft.com/en-us/cli/azure/group?view=azure-cli-latest" target="_blank" rel="noopener noreferrer">Azure CLI documentation</a>.

Create the AKS cluster:

```bash
az aks create --resource-group $AKS_RESOURCE_GROUP \
    --name $TB_CLUSTER_NAME \
    --generate-ssh-keys \
    --enable-addons ingress-appgw \
    --appgw-name $AKS_GATEWAY \
    --appgw-subnet-cidr "10.225.0.0/24" \
    --node-vm-size Standard_D4s_v6 \
    --node-count 3
```

Key parameters:
- `--node-count` — number of nodes (default: 3; adjust with `az aks scale` later).
- `--enable-addons ingress-appgw` — enables the <a href="https://learn.microsoft.com/en-us/azure/application-gateway/" target="_blank" rel="noopener noreferrer">Azure Application Gateway</a> ingress controller.
- `--node-vm-size` — VM size for cluster nodes (default: `Standard_DS2_v2`).
- `--generate-ssh-keys` — generates SSH keys stored in `~/.ssh/`.

Full parameter list: <a href="https://learn.microsoft.com/en-us/cli/azure/aks?view=azure-cli-latest#az_aks_create" target="_blank" rel="noopener noreferrer">az aks create reference</a>.

Alternatively, use the <a href="https://learn.microsoft.com/en-us/azure/aks/learn/quick-kubernetes-deploy-portal?tabs=azure-cli" target="_blank" rel="noopener noreferrer">Azure portal quickstart guide</a>.

## Provision PostgreSQL DB

Set up PostgreSQL on Azure following the <a href="https://learn.microsoft.com/en-us/azure/postgresql/flexible-server/quickstart-create-server-portal" target="_blank" rel="noopener noreferrer">official guide</a> with these requirements:

- PostgreSQL version **16.x**
- Database accessible from the TBMQ cluster
- Initial database name: `thingsboard_mqtt_broker`
- Enable **High availability**

Alternatively, use the `az` CLI (replace `POSTGRESS_USER` and `POSTGRESS_PASS`):

```bash
az postgres flexible-server create --location $AKS_LOCATION --resource-group $AKS_RESOURCE_GROUP \
  --name $TB_DATABASE_NAME --admin-user POSTGRESS_USER --admin-password POSTGRESS_PASS \
  --public-access 0.0.0.0 --storage-size 32 \
  --version 16 -d thingsboard_mqtt_broker
```

Key parameters:
- `--location` — region (from `az account list-locations`)
- `--admin-user` / `--admin-password` — credentials (password: 8–128 chars with uppercase, lowercase, numbers, special chars)
- `--public-access 0.0.0.0` — allows access from all Azure resources; set to `None` to restrict
- `--storage-size` — 32 GiB minimum, 16 TiB maximum
- `--high-availability` — `Disabled` or `Enabled` (only set at creation time)
- `--database-name` (or `-d`) — the name of the database to be created when provisioning the server

Full parameter reference: <a href="https://learn.microsoft.com/en-us/cli/azure/postgres/flexible-server?view=azure-cli-latest#az_postgres_flexible_server_create" target="_blank" rel="noopener noreferrer">az postgres flexible-server create</a>.

Example response:

```text
{

  "databaseName": "thingsboard_mqtt_broker",
```

## Azure Cache for Redis

TBMQ uses Redis to store messages for
<DocLink product={props.product} path="architecture/#persistent-device-client">DEVICE persistent clients</DocLink>
and to reduce database load during authentication. Without caching, every new connection triggers a database query
to validate MQTT client credentials, which can overload the database under high connection rates.

<Aside>
  Starting from **TBMQ v2.1.0**, Redis 7.2.5 is the officially supported version for third-party Redis deployments.
  Only the **Enterprise** and **Enterprise Flash** SKUs of Azure Cache for Redis support Redis 7.2.x.
  The Basic, Standard, and Premium SKUs support only up to **Redis 6.x**. To ensure full compatibility, use an
  Enterprise-tier SKU.
</Aside>

Choose one of the following guides:

- <a href="https://learn.microsoft.com/en-us/azure/azure-cache-for-redis/quickstart-create-redis-enterprise" target="_blank" rel="noopener noreferrer">Quickstart: Create a Redis Enterprise cache</a> **(Recommended)**
- <a href="https://learn.microsoft.com/en-us/azure/azure-cache-for-redis/quickstart-create-redis" target="_blank" rel="noopener noreferrer">Quickstart: Create an open-source Redis cache</a>

For the open-source (legacy) Redis cache, you can also create it using the `az` CLI:

```bash
az redis create --name $TB_REDIS_NAME --location $AKS_LOCATION --resource-group $AKS_RESOURCE_GROUP --sku basic --vm-size C0 --enable-non-ssl-port
```

Key parameters:
- `--name` (or `-n`) — name of the Redis cache
- `--resource-group` (or `-g`) — name of the resource group
- `--sku` — type of Redis cache (accepted values: Basic, Premium, Standard)
- `--vm-size` — size of Redis cache (Basic/Standard: c0–c6; Premium: p1–p5)
- `--location` (or `-l`) — region (from `az account list-locations`)

Full parameter list: <a href="https://learn.microsoft.com/en-us/cli/azure/redis?view=azure-cli-latest" target="_blank" rel="noopener noreferrer">az redis reference</a>.

## Configure Load Balancers

### Configure HTTP(S) Load Balancer

You have two options:

- **HTTP** — no HTTPS support. Suitable for development only.
- **HTTPS** — SSL termination. Recommended for production. Automatically redirects HTTP to HTTPS.

#### HTTP Load Balancer

```bash
kubectl apply -f receipts/http-load-balancer.yml
```

Check provisioning status:

```bash
kubectl get ingress
```

Once ready:

```text
NAME                          CLASS    HOSTS   ADDRESS         PORTS   AGE
tb-broker-http-loadbalancer   <none>   *       34.111.24.134   80      7m25s
```

#### HTTPS Load Balancer

Add a certificate to the Azure Application Gateway:

```bash
az network application-gateway ssl-cert create \
   --resource-group $(az aks show --name $TB_CLUSTER_NAME --resource-group $AKS_RESOURCE_GROUP --query nodeResourceGroup | tr -d '"') \
   --gateway-name $AKS_GATEWAY\
   --name TBMQHTTPSCert \
   --cert-file YOUR_CERT \
   --cert-password YOUR_CERT_PASS
```

Deploy:

```bash
kubectl apply -f receipts/https-load-balancer.yml
```

### Configure MQTT Load Balancer

Create a TCP load balancer that forwards traffic on ports 1883 and 8883:

```bash
kubectl apply -f receipts/mqtt-load-balancer.yml
```

#### MQTT over SSL

Follow <a href="/docs/reference/mqtt-api/getting-connected/#mqtt-over-tls" target="_blank" rel="noopener noreferrer">this guide</a> to create a `.pem` certificate file.
Save it as `server.pem` in the working directory.

Create a ConfigMap from your PEM files:

```bash
kubectl create configmap tbmq-mqtts-config \
 --from-file=server.pem=YOUR_PEM_FILENAME \
 --from-file=mqttserver_key.pem=YOUR_PEM_KEY_FILENAME \
 -o yaml --dry-run=client | kubectl apply -f -
```

Where:
- `YOUR_PEM_FILENAME` — path to your server certificate file
- `YOUR_PEM_KEY_FILENAME` — path to your server certificate private key file

Uncomment all sections marked with "Uncomment the following lines to enable two-way MQTTS" in `tb-broker.yml`:

```bash
kubectl apply -f tb-broker.yml
```

### Upgrade to 2.2.0

This release migrates the MQTT authentication mechanism from YAML/env configuration into the database.
During the upgrade, TBMQ needs to know which authentication providers are enabled — these are provided
as environment variables passed to the upgrade pod.

The upgrade script reads from **database-setup.yml**. Variables from `tb-broker.yml` are
**not** applied during the upgrade — only the values in `database-setup.yml` are used.
Ensure this file reflects your active configuration.

<Aside>
  <ul>
    <li>If you use only Basic authentication, set <code>SECURITY_MQTT_SSL_ENABLED=false</code>.</li>
    <li>If you use only X.509 authentication, set <code>SECURITY_MQTT_BASIC_ENABLED=false</code> and <code>SECURITY_MQTT_SSL_ENABLED=true</code>.</li>
  </ul>
</Aside>

**Supported variables in database-setup.yml:**

- `SECURITY_MQTT_BASIC_ENABLED` (`true|false`)
- `SECURITY_MQTT_SSL_ENABLED` (`true|false`)
- `SECURITY_MQTT_SSL_SKIP_VALIDITY_CHECK_FOR_CLIENT_CERT` (`true|false`) — usually `false`

Once the file is verified, proceed with the <a href="#run-upgrade">upgrade process</a>.
