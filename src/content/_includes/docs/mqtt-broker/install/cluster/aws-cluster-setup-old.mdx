<a href="https://docs.aws.amazon.com/general/latest/gr/aws-sec-cred-types.html" target="_blank" rel="noopener noreferrer">this guide</a>.
The default region should be the ID of the region where you'd like to deploy the cluster:

```bash
aws configure
```

## Configure and create EKS cluster

In the `cluster.yml` file you will find a sample cluster configuration.
You can adjust the following fields according to your requirements:

- `region` — the AWS region where the cluster will be created. Default: `us-east-1`.
- `availabilityZones` — the availability zones within the chosen region. Default: `[us-east-1a, us-east-1b, us-east-1c]`.
- `instanceType` — the EC2 instance type for TBMQ nodes. Default: `m7a.large`.

**Note:** If you don't change `instanceType` and `desiredCapacity`, the EKS will deploy 2 nodes of type `m7a.large`.

<Aside>
  To secure access to PostgreSQL and MSK, configure the existing VPC or create a new one,
  set it as the VPC for TBMQ cluster, create security groups for PostgreSQL and MSK,
  assign them to the <code>managed</code> node group in the TBMQ cluster, and configure access from TBMQ cluster nodes
  to PostgreSQL/MSK using another security group. See the{' '}
  <a href="https://eksctl.io/usage/vpc-networking/" target="_blank" rel="noopener noreferrer">eksctl VPC networking documentation</a> for details.
</Aside>

Create the AWS cluster:

```bash
eksctl create cluster -f cluster.yml
```

## Create an AWS load-balancer controller

Once the cluster is ready, create an AWS load-balancer controller by following
<a href="https://docs.aws.amazon.com/eks/latest/userguide/aws-load-balancer-controller.html" target="_blank" rel="noopener noreferrer">this guide</a>.
The cluster provisioning scripts will create two load balancers:

- `tb-broker-http-loadbalancer` — AWS ALB for the web UI and REST API.
- `tb-broker-mqtt-loadbalancer` — AWS NLB for MQTT communication.

Provisioning the AWS load-balancer controller is a **required step** for these load balancers to work properly.

## Amazon PostgreSQL DB Configuration

Provision a PostgreSQL database on **Amazon RDS**.
One recommended way is to follow the <a href="https://docs.aws.amazon.com/AmazonRDS/latest/UserGuide/CHAP_SettingUp.html" target="_blank" rel="noopener noreferrer">official AWS RDS setup guide</a>.

Recommendations:

- **PostgreSQL version**: Use **16.x**.
- **Template**: Use **Production** for high availability — it enables important settings by default.
- **Parameter group**: Create a custom parameter group for easier future adjustments.
- **Subnets**: Deploy the RDS instance into private subnets to avoid accidental internet exposure.
- **Credentials**: Change the default `username` and set a secure `password`. Store it safely.
- **Connectivity**: Create the database in the **same VPC and subnets** as your TBMQ cluster and assign the
  `eksctl-tbmq-cluster-ClusterSharedNodeSecurityGroup-*` security group.

## Amazon ElastiCache (Redis) Configuration

TBMQ uses Redis to store messages for
<DocLink product={props.product} path="architecture/#persistent-device-client">DEVICE persistent clients</DocLink>
and to reduce database load during authentication. Without caching, every new connection triggers a database query
to validate MQTT client credentials, which can overload the database under high connection rates.

Open the AWS Console → **ElastiCache** → **Redis clusters** → **Create Redis cluster**.

Recommendations:

- **Engine version**: Use Redis **7.x** with a node type of at least **1 GB RAM**.
- **Networking**: Deploy in the **same VPC** as TBMQ, using **private subnets**. Assign the
  `eksctl-tbmq-cluster-ClusterSharedNodeSecurityGroup-*` security group.
- **Backups**: Disable automatic backups.

## Configure Load Balancers

### Configure HTTP(S) Load Balancer

You have two options:

- **HTTP** — no HTTPS support. Suitable for development only.
- **HTTPS** — SSL termination. Recommended for production. Automatically redirects HTTP to HTTPS.

#### HTTP Load Balancer

```bash
kubectl apply -f receipts/http-load-balancer.yml
```

Check provisioning status:

```bash
kubectl get ingress
```

Once ready, output will look like:

```text
NAME                          CLASS    HOSTS   ADDRESS                                                                  PORTS   AGE
tb-broker-http-loadbalancer   <none>   *       k8s-thingsbo-tbbroker-000aba1305-222186756.eu-west-1.elb.amazonaws.com   80      3d1h
```

#### HTTPS Load Balancer

Use <a href="https://aws.amazon.com/certificate-manager/" target="_blank" rel="noopener noreferrer">AWS Certificate Manager</a> to create or import an SSL certificate. Note the certificate ARN.

Replace `YOUR_HTTPS_CERTIFICATE_ARN` in the config:

```bash
nano receipts/https-load-balancer.yml
```

Deploy:

```bash
kubectl apply -f receipts/https-load-balancer.yml
```

### Configure MQTT Load Balancer

Create a TCP load balancer that forwards traffic on ports 1883 and 8883:

```bash
kubectl apply -f receipts/mqtt-load-balancer.yml
```

#### One-way TLS

The most common MQTTS setup is to configure the AWS NLB as a TLS termination point.
Traffic between devices and the load balancer is encrypted; traffic between the load balancer and TBMQ is not.
This is acceptable since the NLB runs within your VPC.

**Limitation:** X.509 certificate client credentials cannot be used with one-way TLS,
since the client certificate is not forwarded to TBMQ.

Use <a href="https://aws.amazon.com/certificate-manager/" target="_blank" rel="noopener noreferrer">AWS Certificate Manager</a> to obtain a certificate. Note the ARN.

Replace `YOUR_MQTTS_CERTIFICATE_ARN` in the config:

```bash
nano receipts/mqtts-load-balancer.yml
```

Deploy:

```bash
kubectl apply -f receipts/mqtts-load-balancer.yml
```

#### Two-way TLS

Two-way TLS allows use of **X.509 certificate** MQTT client credentials.

Follow <a href="/docs/reference/mqtt-api/getting-connected/#mqtt-over-tls" target="_blank" rel="noopener noreferrer">this guide</a> to create a `.pem` certificate file.
Save it as `server.pem` in the working directory.

Create a ConfigMap from your PEM files:

```bash
kubectl create configmap tbmq-mqtts-config \
 --from-file=server.pem=YOUR_PEM_FILENAME \
 --from-file=mqttserver_key.pem=YOUR_PEM_KEY_FILENAME \
 -o yaml --dry-run=client | kubectl apply -f -
```

Where:
- `YOUR_PEM_FILENAME` — path to your server certificate file
- `YOUR_PEM_KEY_FILENAME` — path to your server certificate private key file

Uncomment all sections marked with "Uncomment the following lines to enable two-way MQTTS" in `tb-broker.yml`:

```bash
kubectl apply -f tb-broker.yml
```

Deploy the transparent load balancer:

```bash
kubectl apply -f receipts/mqtt-load-balancer.yml
```

### Upgrade to 2.2.0

This release migrates the MQTT authentication mechanism from YAML/env configuration into the database.
During the upgrade, TBMQ needs to know which authentication providers are enabled — these are provided
as environment variables passed to the upgrade pod.

The upgrade script reads from **database-setup.yml**. Variables from `tb-broker.yml` are
**not** applied during the upgrade — only the values in `database-setup.yml` are used.
Ensure this file reflects your active configuration.

<Aside>
  <ul>
    <li>If you use only Basic authentication, set <code>SECURITY_MQTT_SSL_ENABLED=false</code>.</li>
    <li>If you use only X.509 authentication, set <code>SECURITY_MQTT_BASIC_ENABLED=false</code> and <code>SECURITY_MQTT_SSL_ENABLED=true</code>.</li>
  </ul>
</Aside>

**Supported variables in database-setup.yml:**

- `SECURITY_MQTT_BASIC_ENABLED` (`true|false`)
- `SECURITY_MQTT_SSL_ENABLED` (`true|false`)
- `SECURITY_MQTT_SSL_SKIP_VALIDITY_CHECK_FOR_CLIENT_CERT` (`true|false`) — usually `false`

Once the file is verified, proceed with the <a href="#run-upgrade">upgrade process</a>.

### Upgrade to 2.1.0

TBMQ v2.1.0 introduces the Integration Executor microservice and bumps versions of third-party dependencies.

#### Add Integration Executor microservice

This release adds support for external integrations via the new{' '}
<DocLink product={props.product} path="integrations/">Integration Executor</DocLink> microservice.

To retrieve the latest configuration files, pull the updates from the release branch by following
the <a href="#run-upgrade">run upgrade steps</a> up to (but not including) the upgrade script execution.
The new config files will be included automatically.

The `cluster.yml` file has been updated to include the new managed node group for Integration Executor pods:

```yaml
  - name: tbmq-ie
    instanceType: m7a.large
    desiredCapacity: 2
    maxSize: 2
    minSize: 1
    labels: { role: tbmq-ie }
    ssh:
      allow: true
      publicKeyName: 'your-key-name'
```

Create it with:

```bash
eksctl create nodegroup --config-file=cluster.yml
```

You may skip creating dedicated IE instances. If so, update the `nodeSelector` in `tbmq-ie.yml`
to change the role from `tbmq-ie` to `tbmq` to co-locate IE pods on the same nodes as TBMQ.

#### Update third-party services

TBMQ v2.1.0 updates the following third-party dependencies. See the{' '}
<a href="https://github.com/thingsboard/tbmq/pull/218" target="_blank" rel="noopener noreferrer">pull request</a> for details.

| Service | Previous version | Updated version |
|---|---|---|
| **Redis** | 7.0 | 7.2.5 |
| **PostgreSQL** | 15.x | 16.x |
| **Kafka** | 3.5.1 | 3.7.0 |

<Aside>
  Step-by-step upgrade instructions for third-party managed services are not provided here.
  Consult the official documentation or your service provider's resources.
  After addressing third-party versions, continue with the remaining <a href="#run-upgrade">upgrade steps</a>.
</Aside>

## Cluster deletion

Delete TBMQ nodes:

```bash
./k8s-delete-tbmq.sh
```

Delete all TBMQ nodes and ConfigMaps:

```bash
./k8s-delete-all.sh
```

Delete the EKS cluster (update the cluster name and region if different):

```bash
eksctl delete cluster -r us-east-1 -n tbmq -w
```
