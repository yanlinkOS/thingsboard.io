This tutorial demonstrates how to trigger a related device action based on telemetry received from another device using the Rule Engine.

## Use case

A Smoke Detector reports telemetry: `smoke = "true"`

ThingsBoard:
- creates a **Smoke Alarm**
- finds a related **Fire Alarm System**
- sends an **RPC command** to activate it

When `smoke != "true"`:
- the alarm is cleared

## Architecture

**Devices**
- **Smoke Detector** with the type **Smoke Sensor**
- **Fire Alarm System** with the type **Fire Alarm Device**

**Relation**
- **Smoke Detector** -- **Uses** --> **Fire Alarm System**

**Rule chains**
- modified **Root Rule Chain**
- new **Related Fire Alarm System** rule chain

## Step 3. Create rule chain

You can import a ready-made rule chain or build it manually.

**Option 1. Import rule chain**

- Download the Related Fire Alarm System rule chain as a <a href="/resources/docs/recipes/trigger-related-entities-via-relation/related_fire_alarm_system_rule_chain.json" download="related_fire_alarm_system_rule_chain.json"><strong>JSON file</strong></a> and <DocLink product={props.product} path="user-guide/rule-engine/#import-a-rule-chain" target="_blank">import</DocLink> it into your instance.

**Option 2. Create manually**

- <DocLink product={props.product} path="user-guide/rule-engine/#create-a-rule-chain" target="_blank">Create a new rule chain</DocLink> named `Related Fire Alarm System`, open it, and add the following nodes:

### Step 3.1 Check Relation Presence node

Add the <DocLink product={props.product} path="reference/rule-engine/nodes/filter/check-relation-presence/" target="_blank">Check Relation Presence</DocLink> node and connect it to the **Input** node.

This node will check the relation from the **Fire Alarm System** device to the originator of the message **Smoke Detector** using the type and direction of relation.
If the relation exists, the message will be sent through the **True** chain.

Fill in the fields:
- **Name**: `Check Relation`
- **Relation search parameters**:
  - **Direction**: `To originator`
  - **Relation type**: `Uses`
  - **Check relation to specific entity**: `Fire Alarm System` device

### Step 3.2 Change Originator node

Add the <DocLink product={props.product} path="reference/rule-engine/nodes/transformation/change-originator/" target="_blank">Change Originator</DocLink> node and connect it to the **Check Relation Presence** node with a relation type **True**.

This node will change the originator from the Device **Smoke Detector** to the related Device **Fire Alarm System** and the submitted message will be processed as a message from another entity, namely **Fire Alarm System**.

Fill in the fields:
- **Name**: `Change Originator`
- **New originator**: `Related entity`
- **Relations query**:
  - **Direction**: `From originator`
  - **From originator**: `1`
- **Relation filters**:  
  - **Relation type**: `Uses`
  - **Entity types**: `Device`

### Step 3.3 Script Transformation node

Add the <DocLink product={props.product} path="reference/rule-engine/nodes/transformation/script/" target="_blank">Transformation Script</DocLink> node and connect it to the **Change Originator** node with a relation type **Success**.
Enter the Name field as **New RPC message**.

This node will transform an original message into RPC request message.

The RPC call will have 2 properties:

<table>
  <thead>
    <tr>
      <td><b>Property</b></td><td><b>Value</b></td>
    </tr>
  </thead>
  <tbody>
    <tr>
      <td>method</td>
      <td>ON</td>
    </tr>
    <tr>
      <td>params</td>
      <td>&#123;&#125;</td>
    </tr>
  </tbody>
</table>

In order to do this, add the following Script:

<Tabs>
  <TabItem label="TBEL">
    ```javascript
    newMsg = {};
    if (msg.smoke == 'true') {
    newMsg.method = 'ON';
  }
    newMsg.params = {};
    return {msg: newMsg, metadata: metadata, msgType: msgType};
    ```
  </TabItem>
  <TabItem label="JavaScript">
    ```javascript
    var newMsg = {};
    if (msg.smoke == 'true') {
    newMsg.method = 'ON';
  }
    newMsg.params = {};
    return {msg: newMsg, metadata: metadata, msgType: msgType};
    ```
  </TabItem>
</Tabs>

### Step 3.4 RPC call request node

Add the <DocLink product={props.product} path="reference/rule-engine/nodes/action/rpc-call-request/" target="_blank">RPC call request</DocLink> node and connect it to the **Transformation Script** node with a relation type **Success**.

This node takes the message payload and sends it as a response to the Message Originator **Fire Alarm System**.

Fill in the fields with the input data shown in the following table:
- **Name**: `Fire Alarm System`
- **Fire Alarm System**: `60`

## Step 4. Modify the Root Rule Chain

In the Root Rule Chain, add these nodes after <DocLink product={props.product} path="reference/rule-engine/nodes/action/save-timeseries/" target="_blank">Save Timeseries</DocLink> node:

### Step 4.1 Filter Script node

Add the **Filter Script** node and connect it to the **Save Timeseries** node with a relation type **Success**.

Enter the Name field as **Smoke Alarm Filter**.

This node will check if data of incoming message is **smoke** using the following script:

<Tabs>
<TabItem label="TBEL">
```javascript
return msg.smoke == 'true';
```
</TabItem>
<TabItem label="JavaScript">
```javascript
return msg.smoke == 'true';
```
</TabItem>
</Tabs>

### Step 4.2 Clear Alarm node

Add the **Clear Alarm** node and connect it to the **Filter Script** node with a relation type **False**.

Enter the Name field as **Clear Smoke Alarm** and the Alarm type as **Smoke Alarm**.

This node loads the latest Alarm with configured Alarm Type for Message Originator **Smoke Detector** and Clears the Alarm if it exists.

Add the **Rule Chain** node and connect it to the **Filter Script** node with a relation type **True**.
