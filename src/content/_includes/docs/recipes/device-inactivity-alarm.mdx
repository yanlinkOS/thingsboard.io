Monitor device connectivity and automatically create or clear an alarm when a device stops sending data.

## Use case

**Thermometer A1** (thermostat profile) sends temperature telemetry to ThingsBoard. When it stops reporting data, ThingsBoard marks it inactive and raises a **Device inactive** alarm. When the device reconnects, the alarm clears automatically.

Two approaches are covered:
- **Option 1: Alarm Rule** — no-code, scoped to a device profile. Best for standard inactivity monitoring.
- **Option 2: Rule Engine** — rule chain configuration, supports custom logic such as notifications or external forwarding.

ThingsBoard's <DocLink product={props.product} path="user-guide/device-connectivity-status">Device State service</DocLink> automatically maintains two server attributes on every device:
- `active` — `true` when the device is active, `false` when inactive
- `inactivityAlarmTime` — timestamp of the last inactivity event

## Option 1. Alarm Rule

You can import a ready-made alarm rule or build it manually.

**Option 1. Import alarm rule**

- Download the **Device Active/Inactive** alarm rule as <a href="/resources/docs/recipes/device-inactivity-alarm/device_active_inactive_alarm_rule.json" download="device_active_inactive_alarm_rule.json"><strong>JSON file</strong></a> and <DocLink product={props.product} path="user-guide/alarm-rules/#export--import" target="_blank">import</DocLink> it into your instance.

When importing, set the thermostat as the target device profile.

<Aside type="note">
  During import, set the <strong>thermostat</strong> as the target entity.
</Aside>

**Option 2. Create manually**

Go to **Alarms &#8702; Alarm rules** and click **+ Add alarm rule &#8702; Create new alarm rule**.

## Option 2. Rule Engine

Use this approach when you need custom processing alongside the alarm — for example, sending a notification email, forwarding data to an external system, or chaining conditional logic.

The Rule Engine processes <DocLink product={props.product} path="user-guide/device-connectivity-status">device connectivity events</DocLink> as special message types:
- `INACTIVITY_EVENT` — fired when the device exceeds the inactivity timeout
- `ACTIVITY_EVENT` — fired when the device resumes activity

### Step 1. Set inactivity timeout (optional)

The global default timeout is 600 seconds (`state.defaultInactivityTimeoutInSec` in `thingsboard.yml`).

To override it particular device, add an `inactivityTimeout` server attribute (value in milliseconds):

- **Key**: `inactivityTimeout`
- **Type**: `Integer`
- **Value**: `60000` (60 seconds)

### Step 2. Configure the rule chain

Open your **Root Rule Chain** and create two nodes:

<DocLink product={props.product} path="reference/rule-engine/nodes/action/create-alarm/" target="_blank">Create Alarm</DocLink> node:
- **Name**: `Create Inactivity Alarm`
- **Alarm type**: `Inactivity Timeout`

<DocLink product={props.product} path="reference/rule-engine/nodes/action/clear-alarm/" target="_blank">Clear Alarm</DocLink> node:
- **Name**: `Clear Inactivity Alarm`
- **Alarm type**: `Inactivity Timeout`

Connect these two nodes to the existing <DocLink product={props.product} path="reference/rule-engine/nodes/filter/message-type-switch/" target="_blank">Message Type Switch</DocLink> node:
- **Message Type Switch &#8702;** `Inactivity Event` **&#8702; Create Inactivity Alarm**
- **Message Type Switch &#8702;** `Activity Event` **&#8702; Clear Inactivity Alarm**

<Aside type="note">
  The <strong>Message Type Switch</strong> node is present in the default root rule chain. Add the alarm nodes and connect them to the existing node's <strong>Inactivity Event</strong> and <strong>Activity Event</strong> output links.
</Aside>
